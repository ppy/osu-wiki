## Front matter {#front-matter}

Front matter ต้องวางไว้ที่ส่วนบนสุดของไฟล์เสมอ เขียนด้วยรูปแบบ [YAML](https://en.wikipedia.org/wiki/YAML#Example) และใช้อธิบายข้อมูลเพิ่มเติมเกี่ยวกับบทความ ข้อมูลส่วนนี้จะต้องล้อมรอบด้วยเครื่องหมายขีดสามอัน (`---`) ทั้งด้านบนและด้านล่าง และต้องมีบรรทัดว่างหนึ่งบรรทัดก่อนจะเริ่มหัวข้อชื่อบทความ

## บทความที่ต้องการการปรับปรุง {#articles-that-need-help}

*หมายเหตุ: หลีกเลี่ยงการแปลบทความภาษาอังกฤษที่มีแท็กนี้ นอกจากนี้ ควรเพิ่มแท็กนี้เมื่อบทความแปลนั้นจำเป็นต้องได้รับการจัดระเบียบเนื้อหาใหม่*

แท็ก `needs_cleanup` อาจถูกเพิ่มลงในบทความที่ต้องการการเขียนใหม่หรือการจัดรูปแบบ แท็กนี้ต้องเขียนตามที่แสดงด้านล่าง:

```yaml
needs_cleanup: true
```

เมื่อเพิ่มแท็กนี้ในบทความ ควรเพิ่ม [ความคิดเห็น (Comments)](#comments) เพื่ออธิบายสิ่งที่ต้องทำเพื่อให้สามารถนำแท็กออกได้

## บทความที่ล้าสมัย {#outdated-articles}

*หมายเหตุ: หลีกเลี่ยงการแปลบทความภาษาอังกฤษที่มีแท็กนี้ หากบทความภาษาอังกฤษมีแท็กนี้ บทความแปลก็ต้องมีแท็กนี้ด้วย*

บทความภาษาอังกฤษอาจกลายเป็นบทความที่ล้าสมัยเมื่อเนื้อหาภายในทำให้เข้าใจผิดหรือไม่เกี่ยวข้องอีกต่อไป บทความเหล่านี้ควรได้รับแท็ก `outdated` ซึ่งต้องเขียนดังนี้:

```yaml
outdated: true
```

เมื่อเพิ่มแท็กนี้ในบทความ ควรเพิ่ม [ความคิดเห็น (Comments)](#comments) เพื่ออธิบายสิ่งที่จำเป็นต้องอัปเดตเพื่อให้สามารถนำแท็กออกได้

## บทความแปลที่ล้าสมัย {#outdated-translations}

บทความแปลที่ล้าสมัยต้องใช้แท็ก `outdated_translation` เมื่อบทความต้นฉบับภาษาอังกฤษมีการอัปเดต ยกเว้นการแก้ไขเพียงเล็กน้อยที่ไม่ส่งผลกระทบต่อความหมายของบทความ

```yaml
outdated_translation: true
```

เมื่อระบุว่าบทความแปลล้าสมัย จะต้องเพิ่มแท็ก `outdated_since` ที่ชี้ไปยัง Commit แรกที่มีการอัปเดตบทความเวอร์ชันภาษาอังกฤษด้วย

```yaml
outdated_since: 29eac89cd535f8b071ca000af8fe4f0be22bdc9b
```

แท็กเหล่านี้จะถูกเพิ่มโดยอัตโนมัติหลังจากที่มีการ Merge การเปลี่ยนแปลง ซึ่งสามารถ [ข้ามขั้นตอนได้สำหรับบทความแปลที่ไม่จำเป็นต้องใช้แท็กนี้](/wiki/osu!_wiki/Maintenance#outdated-translations)

## การใส่แท็กบทความ {#tagging-articles}

แท็กช่วยให้ระบบการค้นหาของเว็บไซต์ทำงานได้ดีขึ้น ควรเขียนแท็กด้วยภาษาเดียวกับบทความและรวมรายการแท็กต้นฉบับไว้ด้วย แท็กควรใช้ตัวอักษรพิมพ์เล็กตามความเหมาะสม

ตัวอย่างเช่น บทความชื่อ "Beatmap discussion" อาจมีแท็กดังนี้:

```yaml
tags:
  - beatmap discussions
  - modding V2
  - MV2
```

## บทความแปลที่ยังไม่ผ่านการตรวจสอบ {#translations-without-reviews}

*หมายเหตุ: ผู้ดูแลวิกิ (Wiki maintainers) จะเป็นผู้พิจารณาและใส่แท็กนี้ก่อนทำการ Merge*

ในบางครั้ง บทความแปลถูกเพิ่มเข้ามาในวิกิโดยที่ยังไม่ผ่านการตรวจสอบจากเจ้าของภาษาคนอื่น ในกรณีนี้ จะมีการเพิ่มแท็ก `no_native_review` เพื่อให้ผู้แปลคนอื่นในอนาคตทราบว่าบทความนี้อาจจำเป็นต้องได้รับการตรวจสอบอีกครั้ง:

```yaml
no_native_review: true
```

หรือสามารถใช้แท็ก `no_native_review_since` เพื่อระบุ Commit ที่เริ่มมีการแก้ไขที่ยังไม่ผ่านการตรวจสอบ รวมถึงระบุว่าใครเป็นผู้แก้ไขบทความแปลนั้นตั้งแต่นั้นมา:

```yaml
no_native_review_since: 5539d9e8c943605a7be186dc3f5ab10569275b05 Doryan
```

## การตั้งชื่อบทความ {#article-naming}

*ดูเพิ่มเติมที่: [ชื่อโฟลเดอร์](#folder-names) และ [ชื่อบทความ (Titles)](#titles)*

ชื่อบทความควรเป็นเอกพจน์และใช้รูปแบบ Sentence case (ตัวพิมพ์ใหญ่เฉพาะอักษรแรกของประโยค) ดูรายละเอียดเพิ่มเติมได้ที่ [เกณฑ์การตั้งชื่อของ Wikipedia](https://en.wikipedia.org/wiki/Wikipedia:Naming_conventions_(plurals))

ชื่อบทความควรตรงกับชื่อโฟลเดอร์ที่บทความนั้นอยู่ (สามารถใช้การเว้นวรรคแทนเครื่องหมายขีดล่าง (`_`) ได้ตามความเหมาะสม) หากชื่อโฟลเดอร์เปลี่ยนไป ชื่อบทความควรเปลี่ยนให้ตรงกันด้วย และในทางกลับกัน

---

บทความเกี่ยวกับการประกวดและทัวร์นาเมนต์ถือเป็นข้อยกเว้น โดยชื่อโฟลเดอร์ต้องใช้ตัวย่อ (Abbreviations, Acronyms หรือ Initialisms) แต่ชื่อบทความต้องเป็นชื่อเต็มของการประกวดหรือทัวร์นาเมนต์นั้นๆ

## โครงสร้างโฟลเดอร์และไฟล์ (Folder and file structure) {#folder-and-file-structure}

### ชื่อโฟลเดอร์ (Folder names) {#folder-names}

*ดูเพิ่มเติมที่: [การตั้งชื่อบทความ](#article-naming)*

ชื่อโฟลเดอร์ต้องเป็นภาษาอังกฤษและใช้รูปแบบ Sentence case

ชื่อโฟลเดอร์ต้องใช้ตัวอักษรเหล่านี้เท่านั้น:

- ตัวอักษรพิมพ์ใหญ่และพิมพ์เล็ก
- ตัวเลข
- เครื่องหมายขีดล่าง (`_`)
- เครื่องหมายยัติภังค์ (`-`)
- เครื่องหมายอัศเจรีย์ (`!`)

### ชื่อไฟล์บทความ (Article file names) {#article-file-names}

ชื่อไฟล์ของบทความสามารถดูได้ในคอลัมน์ `ชื่อไฟล์` ของ [ส่วนภาษา](#locales) ตำแหน่งของบทความแปลต้องวางไว้ในโฟลเดอร์เดียวกับบทความภาษาอังกฤษ

## บทความดัชนี {#index-articles}

บทความดัชนีต้องถูกสร้างขึ้นหากโฟลเดอร์นั้นมีไว้เพื่อเก็บเฉพาะบทความอื่นๆ เท่านั้น บทความดัชนีต้องประกอบด้วยรายชื่อบทความที่อยู่ภายในโฟลเดอร์ของตัวเอง และอาจมีข้อมูลอื่นๆ เช่น ย่อหน้านำหรือคำอธิบายของบทความที่ลิงก์ไปหา

## บทความแก้ความกำกวม {#disambiguation-articles}

บทความ [แก้ความกำกวม (Disambiguation)](/wiki/Disambiguation) ต้องวางไว้ในโฟลเดอร์ `/wiki/Disambiguation` และต้องมีการอัปเดตหน้าหลักเพื่อรวมบทความแก้ความกำกวมนั้นไว้ด้วย ดูตัวอย่างได้ที่ [Disambiguation/Mod](/wiki/Disambiguation/Mod)

การเปลี่ยนเส้นทาง (Redirects) ต้องได้รับการอัปเดตเพื่อให้คำสำคัญที่คลุมเครือเปลี่ยนเส้นทางไปยังบทความแก้ความกำกวม

บทความที่ลิงก์มาจากบทความแก้ความกำกวมต้องมี [หมายเหตุส่วนหัวสำหรับการใช้งานอื่น](#for-other-uses) (Hatnote) กำกับไว้

## HTML {#html}

ห้ามใช้ HTML ยกเว้นในส่วนของ [ความคิดเห็น (Comments)](#comments) โครงสร้างของบทความจะต้องถูกจัดระเบียบใหม่หากมีการใช้ HTML

## ความคิดเห็น {#comments}

ควรใช้ความคิดเห็น HTML เพื่อทำเครื่องหมายสิ่งที่ต้องทำ (To-dos) แต่ก็สามารถใช้เพื่อจดบันทึกประกอบข้อความได้เช่นกัน ความคิดเห็นควรอยู่แยกบรรทัดของตัวเอง แต่ก็สามารถวางไว้แทรกในย่อหน้าได้ หากวางแทรกในบรรทัด ส่วนเริ่มต้นของความคิดเห็นต้องไม่มีช่องว่าง

ตัวอย่างที่ไม่ดี:

```markdown
ความคิดเห็น HTML ควรใช้เพื่อทำเครื่องหมายสิ่งที่ต้องทำหรือจดบันทึกประกอบข้อความ
```

ตัวอย่างที่ดี:

```markdown
ความคิดเห็น HTMLควรใช้เพื่อทำเครื่องหมายสิ่งที่ต้องทำหรือจดบันทึกประกอบข้อความ
```

## การแก้ไข {#editing}

### ลำดับการสิ้นสุดบรรทัด {#end-of-line-sequence}

*ข้อควรระวัง: การอัปโหลดไฟล์ Markdown โดยใช้ `CRLF` ผ่าน GitHub จะส่งผลให้ไฟล์เหล่านั้นใช้ `CRLF` เพื่อป้องกันปัญหานี้ ให้ตั้งค่าการสิ้นสุดบรรทัดเป็น `LF` ก่อนทำการอัปโหลด*

ไฟล์ Markdown ต้องถูกตรวจสอบและบันทึกโดยใช้ลำดับการสิ้นสุดบรรทัดแบบ `LF`

### การใช้เครื่องหมาย escape {#escaping}

ไวยากรณ์ Markdown ควรถูก Escape ตามความจำเป็น อย่างไรก็ตาม ชื่อบทความจะถูกประมวลผลเป็นข้อความธรรมดา ดังนั้นจึงห้ามใช้เครื่องหมาย Escape ในชื่อบทความ

### ย่อหน้า {#paragraphs}

แต่ละย่อหน้าต้องตามด้วยบรรทัดว่างหนึ่งบรรทัดเสมอ

### การขึ้นบรรทัดใหม่ {#line-breaks}

การขึ้นบรรทัดใหม่ต้องใช้เครื่องหมาย Backslash (`\`)

ควรใช้การขึ้นบรรทัดใหม่อย่างประหยัด

## หมายเหตุส่วนหัว (Hatnote) {#hatnote}

*อย่าสับสนกับ [ประกาศ (Notice)](#notice)*

Hatnote คือหมายเหตุสั้นๆ ที่วางไว้ที่ส่วนบนสุดของบทความหรือหัวข้อ เพื่อช่วยให้ผู้อ่านไปยังบทความที่เกี่ยวข้อง หรือแจ้งให้ทราบเกี่ยวกับหัวข้อที่เกี่ยวข้องกัน

Hatnote ต้องทำเป็นตัวเอียงและวางไว้ทันทีหลังหัวข้อ หากมีการใช้ Hatnote หลายอัน ต้องวางไว้ในย่อหน้าเดียวกันและแยกกันด้วยการขึ้นบรรทัดใหม่

### หน้าหลัก (Main page) {#main-page}

Hatnote *หน้าหลัก (Main page)* จะนำผู้อ่านไปยังบทความหลักของหัวข้อนั้นๆ เมื่อมีการใช้ Hatnote นี้ จะหมายความว่าหัวข้อที่วางอยู่นั้นเป็นสรุปเนื้อหาของหน้าที่ลิงก์ไปหา Hatnote นี้ควรมีเพียงลิงก์เดียวเท่านั้น และต้องจัดรูปแบบดังนี้:

```markdown
*หน้าหลัก: {บทความ}*

*หน้าหลัก: {บทความ} และ {บทความ}*
```

### ดูเพิ่มเติม (See also) {#see-also}

Hatnote *ดูเพิ่มเติม (See also)* จะแนะนำจุดที่น่าสนใจอื่นๆ จากบทความหรือหัวข้อนั้นๆ ให้กับผู้อ่าน และต้องจัดรูปแบบดังนี้:

```markdown
*ดูเพิ่มเติม: {บทความ}*

*ดูเพิ่มเติม: {บทความ} และ {บทความ}*
```

### สำหรับดู (For see) {#for-see}

Hatnote *สำหรับดู (For see)* จะคล้ายกับ Hatnote *ดูเพิ่มเติม* แต่โดยทั่วไปจะมีการอธิบายและเจาะจงมากกว่า Hatnote นี้อาจใช้ลิงก์มากกว่าหนึ่งลิงก์ได้หากจำเป็น และต้องจัดรูปแบบดังนี้:

```markdown
*สำหรับ {คำอธิบาย}, ดูที่: {บทความ}*

*สำหรับ {คำอธิบาย}, ดูที่: {บทความ} และ {บทความ}*
```

### อย่าสับสนกับ (Not to be confused with) {#not-to-be-confused-with}

Hatnote *อย่าสับสนกับ (Not to be confused with)* ช่วยแยกแยะชื่อบทความหรือหัวข้อที่อาจมีความคลุมเครือหรือเข้าใจผิดได้ Hatnote นี้อาจใช้ลิงก์มากกว่าหนึ่งลิงก์ได้หากจำเป็น และต้องจัดรูปแบบดังนี้:

```markdown
*อย่าสับสนกับ {บทความ}.*

*อย่าสับสนกับ {บทความ} หรือ {บทความ}.*
```

## สำหรับการใช้งานอื่น {#for-other-uses}

Hatnote *สำหรับการใช้งานอื่น (For other uses)* จะคล้ายกับ Hatnote *อย่าสับสนกับ* แต่จะลิงก์โดยตรงไปยัง [บทความแก้ความกำกวม](#disambiguation-articles) Hatnote นี้ต้องลิงก์ไปยังบทความแก้ความกำกวมเท่านั้น และต้องจัดรูปแบบดังนี้:

```markdown
*สำหรับการใช้งานอื่น ดูที่ {บทความแก้ความกำกวม}*
```

## ประกาศ (Notice) {#notice}

*อย่าสับสนกับ [หมายเหตุส่วนหัว (Hatnote)](#hatnote)*

ควรวางประกาศไว้ในที่ที่เหมาะสมภายในหัวข้อ แต่ต้องเริ่มที่ต้นย่อหน้าและใช้ตัวเอียง ประกาศอาจมีการใช้ตัวหนาได้ตามความเหมาะสม แต่ควรใช้ให้น้อยที่สุด ประกาศต้องเขียนเป็นประโยคที่สมบูรณ์ ดังนั้น ประกาศจึงต่างจาก [Hatnote](#hatnote) ส่วนใหญ่ตรงที่ต้องใช้เครื่องหมายจุด (`.`) หรือเครื่องหมายอัศเจรีย์ (`!`) ตามความเหมาะสม และทุกสิ่งที่อยู่ในย่อหน้าเดียวกับประกาศจะต้องทำเป็นตัวเอียงด้วย โดยต้องจัดรูปแบบดังนี้:

```markdown
*หมายเหตุ: {ข้อความ}.*

*ประกาศ: {ข้อความ}.*

*ข้อควรระวัง: {ข้อความ}.*

*คำเตือน: {ข้อความ}.*
```

- `หมายเหตุ (Note)` ควรใช้สำหรับรายละเอียดที่เป็นข้อเท็จจริงหรือข้อมูลปลีกย่อย
- `ประกาศ (Notice)` ควรใช้สำหรับการเตือนความจำหรือเพื่อดึงความสนใจไปยังสิ่งที่ผู้อ่านควรทราบ
- `ข้อควรระวัง (Caution)` ควรใช้เพื่อเตือนผู้อ่านให้หลีกเลี่ยงผลลัพธ์ที่ไม่พึงประสงค์
- `คำเตือน (Warning)` ควรใช้เพื่อเตือนผู้อ่านว่าอาจมีการดำเนินการลงโทษต่อพวกเขาได้

## การวาง Hatnote และประกาศซ้อนกัน (Stacked hatnotes and notices) {#stacked-hatnotes-and-notices}

สามารถวาง Hatnote และประกาศหลายอันซ้อนกันได้หากจำเป็น เมื่อทำเช่นนี้ ต้องวางซ้อนกันโดยไม่มีบรรทัดว่างและใช้เครื่องหมาย Backslash ต่อท้าย:

```markdown
*คำเตือน: {ข้อความ}.*\
*ดูเพิ่มเติม: {บทความ}*
```

ในหลายกรณี การนำ Hatnote หรือประกาศส่วนเกินไปรวมไว้ในเนื้อหาของย่อหน้าแทนที่จะวางซ้อนกันหลายๆ อันอาจมีความเหมาะสมมากกว่า

## การเน้นข้อความ (Emphasising) {#emphasising}

### ตัวหนา (Bolding) {#bolding}

ข้อความตัวหนาต้องใช้เครื่องหมายดอกจันสองอัน (`**`)

ย่อหน้านำอาจทำตัวหนาที่ชื่อบทความซึ่งปรากฏเป็นครั้งแรกได้

### ตัวเอียง (Italics) {#italics}

ข้อความตัวเอียงต้องใช้เครื่องหมายดอกจันอันเดียว (`*`)

คำย่อ (Abbreviation, Acronym หรือ Initialism) ที่ปรากฏครั้งแรกอาจทำเป็นตัวเอียงได้

ตัวเอียงอาจใช้เพื่อเน้นข้อความหรือเพื่อช่วยให้อ่านง่ายขึ้น

ชื่อผลงานหรือวิดีโอเกมควรทำเป็นตัวเอียง ยกเว้นคำว่า osu! (ที่เป็นชื่อเกม)

ตัวอย่างเช่น เมื่ออ้างถึงเพลงในรูปแบบ `{ศิลปิน} - {ชื่อเพลง}` ส่วนทั้งหมดนั้นถือเป็นการอ้างถึงผลงานและควรทำเป็นตัวเอียง:

```markdown
*cYsmix - triangles* เป็นหนึ่งในสามเพลงเริ่มต้นที่สามารถได้ยินได้เมื่อเปิดตัวเกม Client
```

โดยทั่วไปแล้วชื่อศิลปินจะไม่ทำเป็นตัวเอียง ดังนั้นในการอ้างถึงแบบอิสระ ควรทำตัวเอียงเฉพาะชื่อผลงานเท่านั้น เพราะชื่อศิลปินไม่ได้เป็นส่วนหนึ่งของชื่อผลงาน:

```markdown
*Blue Zenith* โดย xi เป็นเพลงที่มีชื่อเสียงในชุมชน osu! เนื่องจากมีคะแนนที่โด่งดังซึ่งทำโดยผู้เล่นระดับท็อปใน Beatmap หนึ่ง
```

### การเน้นข้อความและลิงก์ (Emphasis and links) {#emphasis-and-links}

ข้อความที่เป็นลิงก์จะแสดงเป็นสีที่ต่างออกไปซึ่งถือเป็นการเน้นข้อความอยู่แล้ว ดังนั้นจึงไม่จำเป็นเน้นข้อความเพิ่มเติมอีก:

```markdown
[Camellia - OOPARTS](https://cametek.bandcamp.com/track/parts) เป็นตัวอย่างของเพลงที่สร้างขึ้นเพื่อ osu! อย่างเป็นทางการโดยเฉพาะ หรือที่เรียกว่า *osu! original* เนื่องจากถูกสั่งทำขึ้นพิเศษสำหรับเพลงตัดสิน (Tiebreaker) ในการแข่งขัน osu! World Cup 2020
```

อย่างไรก็ตาม กฎนี้จะไม่ใช้หากผลงานที่อ้างถึงไม่ใช่ส่วนเดียวของลิงก์:

```markdown
[Beatmap ของ Voltaeyx ในเพลง *TheFatRat - Mayday (feat. Laura Brehm)*](https://osu.ppy.sh/beatmapsets/756794) ได้รับความนิยมอย่างมากในปี 2018 เนื่องจากรูปแบบ Slider ที่ซ้อนทับกันอย่างเป็นเอกลักษณ์
```

ควรใช้การจัดรูปแบบลักษณะข้างต้นอย่างประหยัด และห้ามใช้ในจุดที่มีลิงก์จำนวนมาก เช่น ในตารางหรือรายการ (Lists)

## หัวข้อ (Headings) {#headings}

หัวข้อทั้งหมดต้องใช้รูปแบบ Sentence case

หัวข้อต้องใช้รูปแบบ [ATX (Sharp)](https://github.github.com/gfm/#atx-headings) และต้องมีบรรทัดว่างก่อนและหลังหัวข้อเสมอ ยกเว้นหัวข้อที่เป็นชื่อบทความ (Title) ในบรรทัดแรก ซึ่งต้องมีบรรทัดว่างเฉพาะด้านหลังหัวข้อเท่านั้น

หัวข้อต้องมีระดับไม่เกินระดับ 5 และห้ามใช้หัวข้อเพื่อจุดประสงค์ในการจัดรูปแบบข้อความ

### ชื่อบทความ (Titles) {#titles}

*ดูเพิ่มเติมที่: [การตั้งชื่อบทความ](#article-naming))*
*ข้อควรระวัง: ชื่อบทความจะถูกประมวลผลเป็นข้อความธรรมดา ห้ามใช้เครื่องหมาย Escape*

หัวข้อแรกในบทความทั้งหมดต้องเป็นหัวข้อระดับ 1 ซึ่งเป็นชื่อของบทความ หัวข้อหลังจากนั้นทั้งหมดต้องเป็น [หัวข้อส่วน (Section headings)](#sections) ชื่อบทความต้องไม่มีการจัดรูปแบบ ลิงก์ หรือรูปภาพ

หัวข้อชื่อบทความต้องอยู่ในบรรทัดแรกเสมอ ยกเว้นในกรณีที่มีการใช้ [Front matter](#front-matter) ซึ่งหากมีการใช้หัวข้อชื่อบทความจะต้องอยู่ต่อจากนั้นและมีบรรทัดว่างหนึ่งบรรทัดก่อนเริ่มหัวข้อ

### หัวข้อส่วน (Sections) {#sections}

หัวข้อส่วนต้องใช้ระดับ 2 ถึง 5 โดยหัวข้อส่วนที่ต่อจาก [หัวข้อชื่อบทความ](#titles) ต้องเป็นหัวข้อระดับ 2 หัวข้อส่วนสามารถมีไอคอนรูปภาพขนาดเล็กได้ ซึ่งต่างจากหัวข้อชื่อบทความ

หัวข้อส่วนต้องไม่ข้ามระดับหัวข้อ (เช่น ห้ามข้ามจากระดับ 2 ไปเป็นระดับ 4) และต้องไม่มีการจัดรูปแบบหรือลิงก์

*ประกาศ: บนเว็บไซต์ หัวข้อระดับ 4 และ 5 จะไม่ปรากฏในสารบัญ (Table of contents)*

### ตัวระบุที่กำหนดเอง (Custom identifiers) {#custom-identifiers}

สามารถกำหนดตัวระบุ (Identifier) ของหัวข้อใหม่ได้ ซึ่งใช้สำหรับการลิงก์ไปยังหัวข้อนั้นโดยตรง ควรใช้ตัวระบุที่กำหนดเองในกรณีที่ตัวระบุที่สร้างขึ้นโดยอัตโนมัตินั้นยาวเกินไป หรือมีเครื่องหมายวรรคตอนหรือรูปภาพที่จัดการได้ยาก:

```markdown
## ระยะเวลารอคอยของฉันผ่านไปแล้ว ฉันจะอุทธรณ์ได้อย่างไร? {#appeal}

## ตัวอย่างต่างๆ ของเกมเพลย์ osu! {#osu!-gameplay}
```

ฟีเจอร์นี้ยังสามารถใช้สำหรับทำเครื่องหมายส่วนเฉพาะของบทความที่ไม่มีหัวข้อได้ด้วย แต่ควรใช้เครื่องมือนี้อย่างประหยัด:

```markdown
> เรียบร้อย! คุณกำลังอยู่ในเส้นทางสู่การเป็นแชมป์เกมจังหวะ osu! แล้ว!
{#tutorial-quote}
```

## รายการ (Lists) {#lists}

รายการไม่ควรมีการย่อหน้าเกิน 4 ระดับ และไม่ควรมีบรรทัดว่างระหว่างแต่ละรายการ

สำหรับรายการที่ซ้อนกัน เครื่องหมายจุดนำ (Bullets) หรือตัวเลขต้องตรงกับเนื้อหาของรายการหลัก

ตัวอย่างต่อไปนี้ทำผิด (สังเกตการเว้นวรรคก่อนเครื่องหมายจุดนำ):

```markdown
1. เล่นว่าว
  - อย่าเล่นว่าวถ้าฝนตก
```

ตัวอย่างต่อไปนี้ทำถูกต้อง:

```markdown
1. เล่นว่าว
   - อย่าเล่นว่าวถ้าฝนตก
```

### รายการแบบจุดนำ (Bulleted) {#bulleted}

รายการแบบจุดนำต้องใช้เครื่องหมายยัติภังค์ (`-`) และตามด้วยช่องว่างหนึ่งช่องเสมอ (ตามตัวอย่างด้านล่าง)

```markdown
- osu!
  - Hit circle
    - Combo number
    - Approach circle
  - Slider
    - Hit circles
    - Slider body
    - Slider ticks
  - Spinner
- osu!taiko
```

### รายการลำดับเลข (Numbered) {#numbered}

ตัวเลขในรายการลำดับเลขจะต้องเพิ่มขึ้นตามลำดับขั้นตอน

```markdown
1. ดาวน์โหลดตัวติดตั้ง osu!
2. รันตัวติดตั้ง
   1. หากต้องการเปลี่ยนตำแหน่งการติดตั้ง ให้คลิกที่ข้อความใต้แถบความคืบหน้า
   2. ตัวติดตั้งจะถามหาตำแหน่งใหม่ ให้เลือกโฟลเดอร์ที่จะติดตั้ง
3. osu! จะเริ่มทำงานเมื่อการติดตั้งเสร็จสมบูรณ์
4. ลงชื่อเข้าใช้
```

### รายการแบบผสม (Mixed) {#mixed}

การรวมทั้งรายการแบบจุดนำและลำดับเลขควรทำอย่างประหยัด

```markdown
1. ดาวน์โหลดสกินจากฟอรัม
2. โหลดไฟล์สกินเข้าสู่ตัวเกม osu!
   - หากไฟล์เป็นนามสกุล `.zip` ให้แตกไฟล์และย้ายเนื้อหาไปยังโฟลเดอร์ `Skins/` (อยู่ในโฟลเดอร์ที่ติดตั้ง osu!)
   - หากไฟล์เป็นนามสกุล `.osk` ให้เปิดไฟล์นั้นบนเดสก์ท็อปหรือลากและวางลงในตัวเกม
3. เปิด osu! (หากยังไม่ได้เปิด) และเลือกสกินในหน้าตัวเลือก
   - ขั้นตอนนี้อาจเสร็จสิ้นโดยอัตโนมัติหากคุณเปิดไฟล์ `.osk` หรือลากและวางลงในตัวเกม
```

## โค้ด (Code) {#code}

เครื่องหมายสำหรับโค้ดคือ Grave mark (`` ` ``) หากต้องการใส่ Grave mark ในโค้ด ให้ใช้ Grave mark สองอันแทน และหาก Grave mark อยู่ที่จุดเริ่มต้นหรือจุดสิ้นสุด ให้เว้นช่องว่างหนึ่งช่อง (ตามตัวอย่างด้านล่าง)

```markdown
`` ` ``
`` ` Space ` ``
```

### แป้นพิมพ์ (Keyboard keys) {#keyboard-keys}

*ประกาศ: เมื่อต้องการระบุถึงตัวอักษรทั่วไปที่ไม่ใช่แป้นพิมพ์ ให้ใช้เครื่องหมายอัญประกาศแทน*

เมื่อต้องการระบุถึงแป้นพิมพ์ ให้ใช้ตัวอักษรพิมพ์ใหญ่สำหรับอักขระเดี่ยว และใช้รูปแบบ Title case สำหรับแป้นฟังก์ชัน (Modifiers) ใช้สัญลักษณ์บวก (`+`) (โดยไม่ต้องใส่ในรูปแบบโค้ด) เพื่อระบุการกดแป้นพิมพ์ร่วมกัน (ตามตัวอย่างด้านล่าง)

```markdown
pippi สะกดด้วยตัว "p" พิมพ์เล็กเหมือนกับ peppy

กด `Ctrl` + `O` เพื่อเปิดหน้าต่างเปิดไฟล์
```

เมื่อต้องการระบุถึงการเว้นวรรคหรือแป้น Spacebar ให้ใช้ `` `Space` ``

### ข้อความในปุ่มและเมนู (Button and menu text) {#button-and-menu-text}

เมื่อคัดลอกข้อความจากเมนูหรือปุ่ม ควรคัดลอกตัวพิมพ์ใหญ่-เล็กตามที่ปรากฏจริง (ตามตัวอย่างด้านล่าง)

```markdown
ปุ่ม `osu!direct` จะปรากฏในเมนูหลักทางด้านขวา หากคุณมีสถานะ osu!supporter ที่ใช้งานอยู่
```

### ชื่อโฟลเดอร์และไดเรกทอรี (Folder and directory names) {#folder-and-directory-names}

เมื่อคัดลอกชื่อโฟลเดอร์หรือไดเรกทอรี ควรคัดลอกตัวพิมพ์ใหญ่-เล็กตามที่ปรากฏ แต่ควรเลือกใช้เส้นทางตัวพิมพ์เล็กหากเป็นไปได้ เส้นทางไดเรกทอรีต้องไม่เป็นแบบ Absolute (คือห้ามเริ่มชื่อไดเรกทอรีด้วยอักษรระบุไดรฟ์หรือโฟลเดอร์ราก) (ตามตัวอย่างด้านล่าง)

```markdown
โดยปกติแล้ว osu! จะถูกติดตั้งในโฟลเดอร์ `AppData/Local` เว้นแต่จะระบุเป็นอย่างอื่นในระหว่างการติดตั้ง
```

### คำสำคัญและคำสั่ง (Keywords and commands) {#keywords-and-commands}

เมื่อคัดลอกคำสำคัญหรือคำสั่ง ควรคัดลอกตัวพิมพ์ใหญ่-เล็กตามที่ปรากฏหรือตามที่ผู้ใช้ทั่วไปมักจะพิมพ์ และหากทำได้ ให้เลือกใช้ตัวอักษรพิมพ์เล็ก (ตามตัวอย่างด้านล่าง)

```markdown
ในขณะนี้ คำสั่ง `Name` และ `Author` ในไฟล์ตั้งค่าสกิน (`skin.ini`) ยังไม่มีผลใดๆ
```

### ชื่อไฟล์ (File names) {#file-names}

เมื่อคัดลอกชื่อไฟล์ ควรคัดลอกตัวพิมพ์ใหญ่-เล็กตามที่ปรากฏ และหากทำได้ ให้เลือกใช้ตัวอักษรพิมพ์เล็ก (ตามตัวอย่างด้านล่าง)

```markdown
ในการเล่น osu! ให้ดับเบิลคลิกที่ไอคอน `osu!.exe`
```

### นามสกุลไฟล์ (File extensions) {#file-extensions}

*ประกาศ: รูปแบบไฟล์ (ไม่ใช่สิ่งเดียวกับนามสกุลไฟล์) ต้องเขียนด้วยตัวพิมพ์ใหญ่โดยไม่มีเครื่องหมายจุด (`.`) นำหน้า*

นามสกุลไฟล์ต้องขึ้นต้นด้วยเครื่องหมายจุด (`.`) ตามด้วยนามสกุลไฟล์ที่เป็นตัวพิมพ์เล็ก (ตามตัวอย่างด้านล่าง)

```markdown
รูปแบบไฟล์ JPG (หรือ JPEG) จะมีนามสกุลเป็น `.jpg` (หรือ `.jpeg`)
```

### แชนแนลแชท (Chat channels) {#chat-channels}

เมื่อคัดลอกชื่อแชนแนลแชท ให้เริ่มต้นด้วยเครื่องหมายแฮช (`#`) ตามด้วยชื่อแชนแนลที่เป็นตัวพิมพ์เล็ก (ตามตัวอย่างด้านล่าง)

```markdown
`#lobby` คือที่ที่คุณสามารถโฆษณาห้องมัลติเพลเยอร์ของคุณได้
```

## ข้อความจัดรูปแบบล่วงหน้า หรือบล็อกโค้ด (Preformatted text) {#preformatted-text}

*ประกาศ: ปัจจุบันเว็บไซต์ยังไม่รองรับการเน้นไวยากรณ์ (Syntax highlighting) สำหรับข้อความจัดรูปแบบล่วงหน้า*

ข้อความจัดรูปแบบล่วงหน้า (หรือที่เรียกว่า บล็อกโค้ด) ต้องล้อมรอบด้วย Grave mark สามอัน และควรระบุตัวระบุภาษาสำหรับเน้นไวยากรณ์ไว้ด้วย

## ลิงก์ {#links}

*ดูเพิ่มเติมที่: [เชิงอรรถ](#footnotes)*

ลิงก์มีสองประเภทคือ แบบ Inline และแบบ Reference สำหรับแบบ Inline จะมีสองรูปแบบย่อย

ต่อไปนี้คือตัวอย่างของรูปแบบ Inline ทั้งสองแบบ:

```markdown
[ตัวปรับแต่งเกม](/wiki/Gameplay/Game_modifier)

[https://osu.ppy.sh/home](https://osu.ppy.sh/home)
```

ต่อไปนี้คือตัวอย่างของรูปแบบ Reference:

```markdown
[ตัวปรับแต่งเกม][game mods link]

[game mods link]: /wiki/Game_modifier
```

---

ลิงก์ต้องใช้รูปแบบ Inline หากมีการอ้างถึงเพียงครั้งเดียว ควรหลีกเลี่ยงรูปแบบที่ใช้เครื่องหมายวงเล็บแหลม (`<>`) แบบ Inline ส่วนการระบุลิงก์แบบ Reference ต้องวางไว้ที่ส่วนท้ายของบทความ

### ลิงก์ภายใน (Internal links) {#internal-links}

*หมายเหตุ: ลิงก์ภายในหมายถึงลิงก์ที่อยู่ภายในโดเมน `https://osu.ppy.sh/`*

#### ลิงก์ Wiki (Wiki links) {#wiki-links}

ลิงก์ทั้งหมดที่ชี้ไปยังบทความวิกิควรเริ่มต้นด้วย `/wiki/` ตามด้วยเส้นทางของบทความที่คุณต้องการ นอกจากนี้ยังสามารถใช้ลิงก์แบบ Relative ได้ด้วย ตัวอย่างเช่น:

```markdown
[FAQ](/wiki/FAQ)
[pippi](/wiki/Mascots#pippi)
[Beatmaps](../)
[Pattern](./Pattern)
```

ลิงก์วิกิต้องไม่ใช้การเปลี่ยนเส้นทาง (Redirects) และต้องไม่มีเครื่องหมายทับ (`/`) ปิดท้าย

ตัวอย่างที่ไม่ดี:

```markdown
[Article styling criteria](/wiki/ASC)
[Developers](/wiki/Developers/)
[Developers](/wiki/Developers/#game-client-developers)
```

ตัวอย่างที่ดี:

```markdown
[Article styling criteria](/wiki/Article_styling_criteria)
[Developers](/wiki/People/Developers)
[Developers](/wiki/People/Developers#game-client-developers)
```

##### ลิงก์บทความย่อย (Sub-article links) {#sub-article-links}

ลิงก์วิกิที่ชี้ไปยังบทความย่อยควรระบุชื่อโฟลเดอร์ของบทความหลักไว้ในข้อความลิงก์ด้วย ดูตัวอย่างด้านล่าง:

```markdown
*ดูเพิ่มเติมที่: [ตัวแก้ไข Beatmap/การออกแบบ](/wiki/Client/Beatmap_editor/Design)*
```

##### ลิงก์หัวข้อย่อย (Section links) {#section-links}

*ประกาศ: บนเว็บไซต์ หัวข้อระดับ 4 และ 5 จะไม่มีแอตทริบิวต์ ID ทำให้ไม่สามารถลิงก์ไปหาได้โดยตรง*

ลิงก์วิกิที่ชี้ไปยังส่วนต่างๆ ของบทความอาจใช้สัญลักษณ์เครื่องหมายส่วน (`§`) ดูตัวอย่างด้านล่าง:

```markdown
*สำหรับกฎการจับจังหวะ ดูที่: [เกณฑ์การจัดอันดับ § การจับจังหวะ](/wiki/Ranking_criteria#timing)*
```

#### ลิงก์ osu! อื่นๆ (Other osu! links) {#other-osu!-links}

ควรคัดลอก URL จากแถบที่อยู่ของเบราว์เซอร์มาตามจริงเมื่อลิงก์ไปยังหน้าเว็บอื่นๆ ของ osu! โดยต้องคงส่วน `https://osu.ppy.sh` ของ URL ไว้ด้วย

##### โปรไฟล์ผู้ใช้ (User profiles) {#user-profiles}

ชื่อผู้ใช้ทั้งหมดต้องทำเป็นลิงก์เมื่อปรากฏครั้งแรก ส่วนครั้งต่อๆ ไปจะทำหรือไม่ก็ได้ แต่ต้องสม่ำเสมอตลอดทั้งบทความสำหรับทุกชื่อผู้ใช้ หากระบุ ID ผู้ใช้ได้ยาก สามารถข้ามส่วนนี้ไปได้

เมื่อลิงก์ไปยังโปรไฟล์ผู้ใช้ ต้องใช้หมายเลข ID ของผู้ใช้ โดยให้ใช้เว็บไซต์เวอร์ชันใหม่ (`https://osu.ppy.sh/users/{username}`) เพื่อหา ID ของผู้ใช้คนนั้น

ข้อความลิงก์ของผู้ใช้ควรใช้ชื่อปัจจุบันของผู้ใช้

##### ความยาก (Difficulties) {#difficulties}

เมื่อต้องการลิงก์ไปยังความยากใดความยากหนึ่ง ให้ใช้รูปแบบข้อความลิงก์ดังนี้:

```
{ศิลปิน} - {ชื่อเพลง} ({ผู้สร้าง}) [{ชื่อความยาก}]
```

ตัวลิงก์ต้องชี้ไปยังความยากนั้นจริงๆ URL ของความยาก Beatmap ต้องจัดรูปแบบดังนี้:

```
[https://osu.ppy.sh/beatmapsets/](https://osu.ppy.sh/beatmapsets/){BeatmapSetID}#{mode}/{BeatmapID}
```

ชื่อความยากอาจอยู่นอกข้อความลิงก์ได้ แต่ต้องทำอย่างสม่ำเสมอตลอดทั้งบทความ

##### Beatmaps {#beatmaps}

เมื่อต้องการลิงก์ไปยัง Beatmap ให้ใช้รูปแบบข้อความลิงก์ดังนี้:

```
{ศิลปิน} - {ชื่อเพลง} ({ผู้สร้าง})
```

URL ของ Beatmap ทั้งหมดต้องจัดรูปแบบดังนี้:

```
[https://osu.ppy.sh/beatmapsets/](https://osu.ppy.sh/beatmapsets/){BeatmapSetID}
```

### ลิงก์ภายนอก (External links) {#external-links}

*ประกาศ: ลิงก์ภายนอกหมายถึงลิงก์ที่ออกไปนอกโดเมน `https://osu.ppy.sh/`*

ต้องใช้โปรโตคอล `https` เว้นแต่เว็บไซต์นั้นจะไม่รองรับ ลิงก์ภายนอกต้องเป็นลิงก์ที่ตรงไปยังแหล่งข้อมูลที่น่าเชื่อถือ ควรคัดลอก URL จากแถบที่อยู่ของเบราว์เซอร์มาตามจริง

เนื่องจากลิงก์ภายนอกกับลิงก์ osu! ไม่มีความแตกต่างกันในเชิงภาพ ดังนั้นชื่อเว็บไซต์ควรถูกระบุไว้ในข้อความชื่อด้วย ดูตัวอย่างด้านล่าง:

```markdown
*สำหรับข้อมูลเพิ่มเติมเกี่ยวกับทฤษฎีดนตรี ดูที่: [ทฤษฎีดนตรี](https://en.wikipedia.org/wiki/Music_theory)*
```

## รูปภาพ (Images) {#images}

ลิงก์รูปภาพมีสองประเภทคือ แบบ Inline และแบบ Reference ตัวอย่างเช่น:

**รูปแบบ Inline:**

```markdown
![มงกุฎทอง](/wiki/shared/crown-gold.png "อันดับที่ 1")
```

**รูปแบบ Reference:**

```markdown
![มงกุฎทอง][GCrown]

[GCrown]: /wiki/shared/crown-gold.png "อันดับที่ 1"
```

รูปภาพควรใช้รูปแบบ Inline ส่วนการระบุลิงก์แบบ Reference ต้องวางไว้ที่ส่วนท้ายของบทความ

รูปภาพแบบบล็อกทั้งหมดในหน้า (ที่ไม่มีเนื้อหาอื่นอยู่ในบรรทัดเดียวกัน) จะถูกรวมเข้าเป็นแกลเลอรีเดียวกัน ซึ่งสามารถเลื่อนดูได้ด้วยไอคอนลูกศรบนหน้าจอ แป้นพิมพ์ลัด หรือการปัดหน้าจอ

### ข้อความแสดงแทนและหัวข้อรูปภาพ (Alternative and title text) {#alternative-and-title-text}

ข้อความในวงเล็บเหลี่ยมแรก (*Alternative text*) ควรอธิบายรูปภาพตามจริง ซึ่งใช้สำหรับเครื่องอ่านหน้าจอ (Screen readers) หรือเมื่อรูปภาพโหลดไม่ขึ้น สามารถละเว้นได้หากมีเนื้อหาเดียวกับหัวข้อรูปภาพหรือหากรูปภาพนั้นมีไว้เพื่อการตกแต่งเท่านั้น

ข้อความในเครื่องหมายอัญประกาศ (*Title text*) ควรให้ข้อมูลเพิ่มเติมเกี่ยวกับรูปภาพหรือระบุความหมายของรูปภาพ ข้อความนี้จะปรากฏเป็น Tooltip เมื่อวางเมาส์เหนือรูปภาพ และจะใช้เป็นคำบรรยายภาพในกรณีที่เหมาะสม โดยข้อความนี้ไม่รองรับการจัดรูปแบบ Markdown ใดๆ

### การแสดงผล (Display) {#display}

หากรูปภาพเป็นเนื้อหาเพียงอย่างเดียวในย่อหน้า จะถูกแสดงผลเป็นบล็อกกึ่งกลาง มิฉะนั้นจะไหลไปตามข้อความ Inline โดยรอบ

รูปภาพแบบบล็อกที่มีหัวข้อรูปภาพจะแสดงหัวข้อนั้นเป็นคำบรรยายใต้ภาพ หลีกเลี่ยงการเพิ่ม [ความคิดเห็น HTML](#comments) หรือข้อความอื่นๆ ในบรรทัดเดียวกับรูปภาพ เพราะจะทำให้คำบรรยายใต้ภาพไม่แสดงผล

รูปภาพแบบบล็อกมักจะใช้คู่กับการจัดรูปแบบ [Infobox](#infoboxes) เพื่อลดขนาดเริ่มต้นและทำให้ลอยอยู่ด้านข้างของเนื้อหาอื่น:

```markdown
::: Infobox
![](img/mod-response.png "ตัวอย่างการตอบกลับการ Mod")
:::
```

### การแคชรูปภาพ (Image caching) {#image-caching}

รูปภาพบนเว็บไซต์จะถูกแคชไว้นานสูงสุด 60 วัน โดยรูปภาพที่แคชไว้จะจับคู่กับ URL ของลิงก์รูปภาพ

เมื่อมีการอัปเดตรูปภาพ ให้เปลี่ยนชื่อรูปภาพหรือเพิ่ม Query string ต่อท้าย URL ทั้งนี้ควรประสานงานให้อัปเดตลิงก์รูปภาพในบทความแปลทั้งหมดด้วย

### รูปแบบและคุณภาพ (Formats and quality) {#formats-and-quality}

รูปภาพควรใช้รูปแบบ JPG ที่คุณภาพระดับ 8 (80 หรือ 80% ขึ้นอยู่กับโปรแกรม) หากรูปภาพมีความโปร่งใสหรือมีข้อความที่ต้องอ่านให้ชัดเจน ให้ใช้รูปแบบ PNG แทน หากรูปภาพเป็นภาพเคลื่อนไหว สามารถใช้รูปแบบ GIF ได้ แต่ควรใช้อย่างประหยัดเพราะโหลดนานและอาจเกิน [ขนาดไฟล์สูงสุด](#file-size)

### ขนาดไฟล์ {#file-size}

รูปภาพต้องมีขนาดต่ำกว่า 1 เมกะไบต์ มิฉะนั้นจะไม่สามารถโหลดได้ การลดขนาดภาพและใช้ JPG ที่คุณภาพ 80% มักจะช่วยให้ขนาดไฟล์ไม่เกินกำหนด

รูปภาพทั้งหมดควรได้รับการปรับแต่งให้มีขนาดเหมาะสมที่สุด โดยแนะนำให้ใช้ [jpeg-archive](https://github.com/danielgtaylor/jpeg-archive) ในการบีบอัดรูปภาพ JPEG และเพื่อความสม่ำเสมอ ให้ใช้คำสั่งดังนี้:

```
jpeg-recompress -am smallfry <input> <output>
```

นอกจากนี้ยังสามารถใช้ [ezgif.com](https://ezgif.com/) ในการแปลงและปรับแต่งรูปภาพได้

### ชื่อไฟล์ (File names) {#file-names}

*ประกาศ: นามสกุลไฟล์ต้องใช้ตัวพิมพ์เล็ก มิฉะนั้นจะไม่สามารถโหลดได้!*

ใช้เครื่องหมายยัติภังค์ (`-`) ในการเว้นวรรคคำ เมื่อตั้งชื่อรูปภาพ ชื่อไฟล์ควรมีความหมายหรือสื่อถึงรูปภาพแต่ยังคงความสั้น

รูปภาพต้องถูกเก็บไว้ในโฟลเดอร์ชื่อ `img` ภายใต้โฟลเดอร์ของบทความ ส่วนรูปภาพที่ใช้ในหลายบทความควรเก็บไว้ในโฟลเดอร์ `/wiki/shared`

### ความกว้างของรูปภาพสูงสุด {#max-image-width}

ความกว้างสูงสุดของรูปภาพบนเว็บไซต์จะเท่ากับความกว้างของเนื้อหาบทความ โดยรูปภาพไม่ควรกว้างเกิน 800 พิกเซล

### การใส่คำบรรยายในรูปภาพ (Annotating images) {#annotating-images}

เมื่อใส่คำบรรยายในรูปภาพ ให้ใช้ฟอนต์ *Torus Regular* สำหรับตัวอักษรจีน เกาหลี และญี่ปุ่น ให้ใช้ฟอนต์ *Microsoft YaHei*

ควรหลีกเลี่ยงการใส่คำบรรยายในรูปภาพ เพราะจะทำให้นักแปล (และบรรณาธิการคนอื่นๆ) แก้ไขได้ยาก

#### การแปลรูปภาพที่มีคำบรรยาย (Translating annotated images) {#translating-annotated-images}

เมื่อแปลรูปภาพที่มีคำบรรยาย รูปภาพเวอร์ชันที่แปลแล้วต้องวางไว้ในโฟลเดอร์เดียวกับเวอร์ชันดั้งเดิม (เช่น เวอร์ชันภาษาอังกฤษ) ชื่อไฟล์ของรูปภาพที่แปลแล้วต้องขึ้นต้นด้วยชื่อเวอร์ชันดั้งเดิม ตามด้วยเครื่องหมายยัติภังค์ และตามด้วยชื่อ Locale (เป็นตัวพิมพ์ใหญ่) ดูตัวอย่างด้านล่าง:

- `hardrock-mod-vs-easy-mod.jpg` สำหรับภาษาอังกฤษ
- `hardrock-mod-vs-easy-mod-DE.jpg` สำหรับภาษาเยอรมัน
- `hardrock-mod-vs-easy-mod-ZH-TW.jpg` สำหรับภาษาจีนตัวเต็ม

### ภาพหน้าจอการเล่นเกม {#screenshots-of-gameplay}

ภาพหน้าจอการเล่นเกมทั้งหมดต้องถ่ายจากตัวเกมเวอร์ชัน Stable เว้นแต่จะเป็นฟีเจอร์เฉพาะที่ไม่มีในเวอร์ชัน Stable และควรใช้ฟีเจอร์ถ่ายภาพหน้าจอภายในเกม (`F12`)

#### การตั้งค่าตัวเกม (Game client settings) {#game-client-settings}

*หมายเหตุ: หากคุณไม่ต้องการเปลี่ยนการตั้งค่าปัจจุบันสำหรับวิกิ คุณสามารถย้ายไฟล์ `osu!.<ComputerUser>.cfg` ออกจากโฟลเดอร์ osu! ก่อนแล้วค่อยย้ายกลับมาในภายหลัง*

คุณต้องตั้งค่าต่อไปนี้ก่อนถ่ายภาพหน้าจอตัวเกม (การตั้งค่าที่ไม่ได้ระบุไว้ด้านล่างนี้ให้ใช้ค่าเริ่มต้น):

- ภาษา (Select language): `English`
- แสดงข้อมูลเพลงในภาษาต้นฉบับ (Prefer metadata in original language): `เปิดใช้งาน`
- ความละเอียด (Resolution): `1280x720`
- โหมดเต็มหน้าจอ (Fullscreen mode): `ปิดใช้งาน`
- พารัลแลกซ์ (Parallax): `ปิดใช้งาน`
- เคล็ดลับเมนู (Menu tips): `ปิดใช้งาน`
- พื้นหลังตามฤดูกาล (Seasonal backgrounds): `ไม่เคย`
- แสดงแป้นพิมพ์จำลองเสมอ (Always show key overlay): `เปิดใช้งาน`
- สกินปัจจุบัน (Current skin): `Default` (ตัวเลือกแรก)

*ประกาศสำหรับนักแปล: หากคุณกำลังแปลบทความที่มีภาพหน้าจอตัวเกม คุณสามารถตั้งค่าภาษาของตัวเกมเป็นภาษาที่คุณกำลังแปลได้*

### ลิงก์รูปภาพ (Image links) {#image-links}

รูปภาพต้องไม่เป็นส่วนหนึ่งของข้อความลิงก์

## ไอคอนธง (Flag icons) {#flag-icons}

ไอคอนธงใช้รหัสสองตัวอักษร (เป็นตัวพิมพ์ใหญ่ทั้งหมด) ให้ตรงกับดินแดนนั้นๆ เมื่อต้องการเพิ่มธงแบบ Inline ให้ใช้รูปแบบดังนี้:

```markdown
::{ flag=XX }::
```

โดยที่ `XX` คือรหัสประเทศสองตัวอักษรตามมาตรฐาน [ISO 3166-1 alpha-2](https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2)

## ตาราง (Tables) {#tables}

ตารางบนเว็บไซต์รองรับเฉพาะหัวข้อในแถวแรกเท่านั้น

ห้ามตกแต่งตารางให้สวยงามด้วยการเพิ่มช่องว่างเพื่อจัดความกว้างให้เท่ากัน ตารางต้องมีเส้นแนวตั้ง (`|`) ทั้งด้านซ้ายและขวา และข้อความในแต่ละเซลล์ต้องมีการเว้นวรรคหนึ่งช่องทั้งสองด้าน เซลล์ว่างให้ใช้เส้นแนวตั้ง (`|`) ตามด้วยช่องว่างสองช่องและเส้นแนวตั้งอีกอัน (`|`)

แถวตัวคั่น (บรรทัดถัดจากหัวข้อตาราง) ต้องใช้เพียงสามอักขระต่อคอลัมน์ (และเว้นวรรคหนึ่งช่องทั้งสองด้าน) โดยต้องมีลักษณะอย่างใดอย่างหนึ่งดังนี้:

- `:--` (สำหรับจัดชิดซ้าย)
- `:-:` (สำหรับจัดกึ่งกลาง)
- `--:` (สำหรับจัดชิดขวา)

---

ต่อไปนี้คือตัวอย่างของตารางที่ถูกต้อง:

```markdown
| Team "Picturesque" Red | Score | Team "Statuesque" Blue | Average Beatmap Stars |
| :-- | :-: | --: | :-- |
| **peppy** | 5 - 2 | pippi | 9.3 stars |
| Aiko | 1 - 6 | **Alisa** | 4.2 stars |
| Ryūta | 3 - 4 | **Yuzu** | 5.1 stars |
| **Taikonator** | 7 - 0 | Tama | 13.37 stars |
| Maria | No Contest | Mocha |   |
```

## กล่องข้อมูล (Infoboxes) {#infoboxes}

กล่องข้อมูล (Infobox) คือบล็อกที่มีความกว้างคงที่ซึ่งจะจัดชิดทางด้านขวาของบทความ อาจประกอบด้วยรูปภาพที่เกี่ยวข้องซึ่งช่วยอธิบายเนื้อหาโดยรอบ หรือเป็นบล็อกนำทางที่ลิงก์ไปยังบทความอื่นๆ ในหมวดหมู่เดียวกัน

ตัวอย่างการใช้งานซึ่งจะแสดงผลทางด้านขวา (หรือด้านบนสำหรับหน้าจอขนาดเล็ก):

::: Infobox
![](/wiki/shared/mods/SD.png "ไอคอนม็อด Sudden Death")
:::

```markdown
::: Infobox
![](/wiki/shared/mods/SD.png "ไอคอนม็อด Sudden Death")
:::
```

ควรใช้ Infobox อย่างระมัดระวังในกรณีต่อไปนี้:

- หัวข้อที่สั้นเกินไป: หัวข้อถัดไปจะไปปรากฏใต้ Infobox ทำให้เกิดช่องว่างขนาดใหญ่หลังข้อความ
- มีรูปภาพหลายรูปพร้อมกัน: แนะนำให้ใช้ Infobox แยกกันเพื่อการออกแบบที่ดีกว่า

ภายใน Infobox อนุญาตให้ใช้เฉพาะหัวข้อระดับ 4 และ 5 เท่านั้น เนื่องจากจะไม่ปรากฏในสารบัญและมีขนาดที่เหมาะสม หากการใช้หัวข้อระดับเหล่านี้ทำให้เกิด [ข้อผิดพลาดจากการตรวจสอบ CI](/wiki/osu!_wiki/Maintenance#ci-checks) อาจจำเป็นต้องทำการข้ามขั้นตอนการตรวจสอบนั้นไป

## เชิงอรรถ (Footnotes) {#footnotes}

เชิงอรรถคือหมายเหตุสั้นๆ ที่อยู่ส่วนท้ายของหน้า ใช้สำหรับอ้างอิงแหล่งที่มา หรือให้ข้อมูลเบื้องหลังที่จะรบกวนการไหลลื่นของบทความ เชิงอรรถสามารถมีการจัดรูปแบบข้อความและลิงก์ได้

ใน osu! wiki เชิงอรรถจะใช้ไวยากรณ์พิเศษ (`[^identifier]`) สามารถใช้ตัวระบุใดก็ได้ แต่จะถูกแสดงผลเป็นตัวยกด้วยตัวเลขที่เพิ่มขึ้นตามลำดับที่ปรากฏครั้งแรก บทความแปลห้ามทำการแก้ไขตัวระบุของเชิงอรรถ

การอ้างอิงเชิงอรรถจะวางไว้ต่อท้ายคำ วลี หรือประโยคที่ต้องการอธิบายโดยไม่มีการเว้นวรรค และต้องวางไว้หลังเครื่องหมายวรรคตอน ยกเว้นวงเล็บ (หากเกี่ยวเนื่องกับเนื้อหาภายในวงเล็บ) และเครื่องหมายขีด

เชิงอรรถเองต้องวางไว้ภายใต้หัวข้อระดับสองที่แยกออกมาต่างหากที่ส่วนท้ายของบทความ โดยขึ้นอยู่กับเนื้อหา หัวข้อที่ใช้อาจเป็น `อ้างอิง (References)`, `หมายเหตุ (Notes)` หรือ `หมายเหตุและอ้างอิง (Notes and references)`

ตัวอย่างการใช้งานที่ถูกต้อง:

```markdown
osu! wiki คือโปรเจกต์ที่มีเป้าหมายเพื่อแทนที่ระบบ FAQ เดิม[^wiki-faq] ชื่อของมันถูกตั้งตามเกมจังหวะ osu![^osu] และสารานุกรมออนไลน์ที่ใหญ่ที่สุดอย่าง Wikipedia ตั้งแต่เริ่มต้น มันได้ดึงดูดเหล่านักแปล[^wiki-tl] และบรรณาธิการที่มีความสามารถมากมาย
```

### การอ้างอิง (Citations) {#citations}

การอ้างอิงใช้สำหรับระบุแหล่งที่มาของข้อมูล ควรเลือกใช้การอ้างอิงผ่านเชิงอรรถมากกว่าลิงก์แบบ Inline

การอ้างอิงควรระบุชื่อผู้แต่ง, วันที่, บริการ/แพลตฟอร์ม และชื่อเรื่องตามความเหมาะสม รูปแบบอาจแตกต่างกันไปตามแหล่งข้อมูลแต่เน้นความกระชับ

ตัวอย่าง:

```markdown
osu!api เวอร์ชันแรกเปิดให้ใช้งานเมื่อวันที่ 2 กรกฎาคม 2013[^api-first-usage] และได้รับการชื่นชมอย่างมากจากผู้ใช้[^api-praise] เวอร์ชันใหม่ของ API ที่เปิดตัวหลายปีต่อมามีความสามารถเพิ่มขึ้นอย่างมาก[^api-v2-2020] การระบุเวอร์ชันของ Endpoint เป็นเรื่องปกติสำหรับเว็บ API[^web-api]

[^api-praise]: [โพสต์ฟอรัมโดย Menchi (2013-11-02) ในกระทู้ "osu!api open beta"](https://osu.ppy.sh/community/forums/posts/2662247)
```

### หมายเหตุ (Notes) {#notes}

เชิงอรรถอาจใช้สำหรับเก็บคำอธิบายหรือหมายเหตุปลีกย่อยที่ไม่สามารถใส่ไว้ในเนื้อหาได้โดยตรงเพื่อไม่ให้เสียอรรถรสในการอ่าน หรือมีความสำคัญน้อยกว่าเนื้อหาหลักของบทความ เชิงอรรถประเภทนี้สามารถใช้ข้อความแบบอิสระได้

ตัวอย่าง:

```markdown
ทัวร์นาเมนต์ต้องไม่ถูกจัดและดำเนินการโดยทีมงานที่ขาดประสบการณ์และขาดชื่อเสียงที่น่าเชื่อถือ[^staff]

## หมายเหตุ {#notes.1}

[^staff]: ทีมงานที่ *ขาดประสบการณ์* ถูกให้คำนิยามอย่างกว้างๆ คือผู้ที่เล่น osu! รวมกันน้อยกว่าหนึ่งชั่วโมง
```

## การยกข้อความ (Blockquotes) {#blockquotes}

การใช้ Blockquote ถูกจำกัดไว้เพียงเพื่อ [การอ้างอิงคำพูดของใครบางคนหรือบางสิ่ง](/wiki/Article_styling_criteria/Writing#block-quotation) เท่านั้น ห้ามใช้เพื่อจัดรูปแบบข้อความเพื่อจุดประสงค์อื่น

```markdown
> plz enjoy game

—rrtyui
```

## เส้นคั่นเนื้อหา (Thematic breaks) {#thematic-breaks}

เส้นคั่นเนื้อหา (หรือที่เรียกว่า เส้นแนวนอน) ควรใช้อย่างประหยัด ตัวอย่างการใช้งานเส้นคั่นเนื้อหา ได้แก่ (แต่ไม่จำกัดเพียง):

- ใช้แยกรูปภาพออกจากข้อความ
- ใช้แยกรูปภาพหลายรูปที่วางต่อกัน
- ใช้เปลี่ยนหัวข้อภายในส่วนเดียวกัน

ต้องมีบรรทัดว่างก่อนและหลังเส้นคั่นเสมอ เส้นคั่นเนื้อหาต้องใช้เครื่องหมายยัติภังค์เพียงสามตัวเท่านั้น ดังนี้:

```markdown
---
```
