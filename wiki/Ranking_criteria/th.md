# Ranking criteria

บทความนี้กำหนดกฎและแนวทางที่ [บีตแมป](/wiki/Beatmap) ต้องทำตาม เพื่อให้ผ่าน [ขั้นตอนการจัดอันดับบีตแมป](/wiki/Beatmap_ranking_procedure)

- **กฎห้ามละเมิดไม่ว่าในกรณีใดก็ตาม**
- **แนวทางสามารถละเว้นหรือฝ่าฝืนได้ในสถานการณ์พิเศษ** ในกรณีแบบนั้น แมปเปอร์ควรให้คำอธิบายที่เพียงพอไว้ล่วงหน้า หรือเมื่อมีคนถามระหว่างขั้นตอนม็อดดิ้ง

## สารบัญ

บางส่วนของ ranking criteria มีบทความแยกของตัวเอง แต่ยังมีผลกับทุกบีตแมป ไม่ว่าจะเป็นโหมดเกม ความยาว หรือข้อจำกัดอื่นใดก็ตาม

### สรุป

- **[Simplified ranking criteria](Simplified_ranking_criteria):** บทความนี้สรุปภาพรวมใหญ่ ๆ ของบทความ ranking criteria เพราะชุดบทความนี้ค่อนข้างซับซ้อน โปรดทราบว่าบทความนี้ไม่ได้ครอบคลุมทุกอย่าง และควรอ้างอิงบทความ ranking criteria ที่เกี่ยวข้องเมื่อไม่แน่ใจ

### ส่วนขยายทั่วไป

- **[Content usage permissions](/wiki/Rules/Content_usage_permissions)**: แนวทางการใช้สื่อสำหรับทุกบีตแมป
- **[Visual content considerations](/wiki/Rules/Visual_content_considerations)**: แนวทางเกี่ยวกับภาพทุกชนิดที่พบในพื้นหลัง สตอรีบอร์ด หรือวิดีโอของบีตแมป osu!
- **[Song content rules](/wiki/Rules/Song_content_rules)**: แนวทางเกี่ยวกับคอนเทนต์เสียงทุกชนิดในบีตแมป
- **[Code of conduct for modding and mapping](/wiki/Rules/Code_of_conduct_for_modding_and_mapping)**: กฎและแนวทางสำหรับการมีส่วนร่วมในขั้นตอนม็อดดิ้งบีตแมป
- **[Beatmap ranking procedure](/wiki/Beatmap_ranking_procedure)**: ขั้นตอนที่บีตแมปทุกอันต้องผ่านเพื่อให้ได้ Ranked

### Ranking criteria ทั่วไป

- **[Metadata](Metadata)**: กฎและแนวทางเกี่ยวกับการให้เครดิตในบีตแมป

### Ranking criteria เฉพาะโหมดเกม

นอกจากนี้ แต่ละ [โหมดเกม](/wiki/Game_mode) ยังมี ranking criteria เฉพาะที่ใช้กับบีตแมปของโหมดนั้นเท่านั้น

- **[osu!](osu!)**
- **[osu!taiko](osu!taiko)**
- **[osu!catch](osu!catch)**
- **[osu!mania](osu!mania)**

## การเสนอการเปลี่ยนแปลง

การเปลี่ยนแปลงใด ๆ ใน ranking criteria จะเสนอผ่าน [ฟอรัม Ranking Criteria](https://osu.ppy.sh/community/forums/87)

1. **สร้างข้อเสนอใน [ฟอรัม Ranking Criteria](https://osu.ppy.sh/community/forums/87)** พยายามอธิบายไอเดียและผลกระทบของการเปลี่ยนแปลงให้ละเอียดที่สุดเท่าที่ทำได้
2. **หาข้อสรุปร่วมกัน** หากคุณคิดว่าการพูดคุยถึงจุดนั้นแล้ว ให้ติดต่อสมาชิก [NAT](/wiki/People/Nomination_Assessment_Team) เพื่อดูว่าข้อเสนอของคุณสามารถนำไปใช้ได้หรือไม่ ในระดับหนึ่ง NAT สามารถทำหน้าที่เป็นผู้ตัดสินในกรณีที่มีข้อโต้แย้งได้ แต่ข้อเสนอที่ไม่มีฉันทามติโดยทั่วไปมักจะไม่ผ่าน
3. **สร้าง pull request บน [osu! wiki GitHub repository](https://github.com/ppy/osu-wiki)** โดยปกติแล้วขั้นตอนนี้จะทำโดยสมาชิก NAT ที่เกี่ยวข้องกับการสรุปการพูดคุย
4. **รอการอนุมัติจาก [peppy](https://osu.ppy.sh/users/2)**

การเปลี่ยนแปลงที่ไม่ได้เปลี่ยนความหมายหรือเจตนาของ ranking criteria เช่น การแก้สะกด ไวยากรณ์ หรือการจัดระเบียบ อาจไม่จำเป็นต้องมีข้อเสนอในฟอรัม อย่างไรก็ตาม ยังแนะนำให้ตรวจสอบกับสมาชิกชุมชนที่มีประสบการณ์ว่าการเปลี่ยนแปลงนั้นมีประโยชน์และเหมาะสม

## เครื่องมือ

ขอแนะนำอย่างยิ่งให้ใช้ [Mapset Verifier (MV)](https://github.com/Naxesss/MapsetVerifier) เพื่อช่วยตรวจ criteria เหล่านี้ [AiMod](/wiki/Client/Beatmap_editor/AiMod) จากไคลเอนต์เก่า (stable) ล้าสมัยและไม่มีการดูแลต่อแล้ว ส่วนระบบตรวจบีตแมปใน [osu!(lazer)](/wiki/Client/Release_stream/Lazer) ยังอยู่ในสถานะ [กำลังพัฒนา](https://github.com/ppy/osu/issues/12091#issuecomment-878760791)

สิ่งสำคัญที่ควรเข้าใจก่อนใช้งาน:

- ไม่มีเครื่องมือไหนแทนการอ่านหน้านี้หรือการตรวจด้วยตัวเองได้ทั้งหมด
- ควรพิจารณาสิ่งที่เครื่องมือแจ้งอย่างมีวิจารณญาณ และอย่าทำตามแบบไม่คิด
- เครื่องมือมีไว้ช่วยคุณ ไม่ใช่มาแทนคุณ

## ทั่วไป

### กฎ

- **ห้ามวาง hit object สองอันบน tick เดียวกัน** ซึ่งรวมถึงฮิตเซอร์เคิล รวมถึงระยะเวลาของสไลเดอร์และสปินเนอร์ด้วย ยกเว้น difficulty ของ osu!mania
- **ต้องมีระยะห่างอย่างน้อย 10 มิลลิวินาทีระหว่างฮิตเซอร์เคิลกับ object ถัดไป และอย่างน้อย 20 มิลลิวินาทีระหว่างท้ายสไลเดอร์กับ object ถัดไป** ยกเว้น difficulty ของ osu!mania และ osu!catch
- **คอนเทนต์ทั้งหมดในบีตแมปต้องได้รับการเคลียร์ตาม [content usage permissions](/wiki/Rules/Content_usage_permissions#artist-permissions)** สำหรับศิลปินหรือแทร็กที่ถูกระบุว่าไม่อนุญาตในหน้านั้น เพลงของพวกเขาจะจัดอันดับได้ก็ต่อเมื่อ beatmap host ได้รับอนุญาตสำหรับแทร็กนั้นโดยชัดเจน ต้องแสดงหลักฐานการอนุญาตไว้ในคำอธิบายบีตแมปหรือหน้าพูดคุย
- **hit object, hitsound และ timing ของบีตแมปต้องสร้างจากอินพุตของมนุษย์โดยตรงเท่านั้น ห้ามใช้เครื่องมือ generative ใด ๆ** การสร้างบีตแมปเป็นกระบวนการสร้างสรรค์โดยพื้นฐาน ดังนั้นการใช้ทางลัดอย่าง generative AI จึงไม่เป็นที่ยอมรับสำหรับการจัดอันดับ
- **ห้ามมีภาพลามกอนาจารในคอนเทนต์ [พื้นหลัง](/wiki/Beatmap/Background)/สตอรีบอร์ด/วิดีโอ** ซึ่งรวมถึงภาพเปลือย กึ่งเปลือย การสื่อทางเพศ ความรุนแรง การใช้ยาเสพติด ฯลฯ ดูกฎแบบละเอียดได้ที่ [Visual Content Considerations](/wiki/Rules/Visual_content_considerations)
- **Difficulty ที่มีแสงกะพริบซ้ำ ๆ ภาพกระพริบเป็นจังหวะ หรือการเปลี่ยนคอนทราสต์ ความสว่าง หรือสีอย่างรวดเร็วในสตอรีบอร์ดหรือวิดีโอ ต้องใช้คำเตือนโรคลมชัก** หากคำเตือนรบกวน gameplay ต้องเพิ่ม audio lead-in ให้ยาวขึ้น เอฟเฟกต์กะพริบที่ 3 Hz (เช่น 3 ครั้งต่อวินาที) หรือต่ำกว่านั้นมักไม่น่ากังวล หากไม่แน่ใจ ให้ใส่คำเตือนและยืนยันความจำเป็นระหว่างขั้นตอนม็อดดิ้ง
- **ห้ามมีไฟล์ที่ไม่ได้ใช้หรือไฟล์ขนาด 0 ไบต์ในโฟลเดอร์บีตแมป** ไฟล์ 0 ไบต์ทำให้ไฟล์อื่นในโฟลเดอร์บีตแมปอัปโหลดได้ไม่ถูกต้อง ข้อยกเว้นเดียวคือไฟล์ `thumbs.db` ที่สร้างขึ้นอัตโนมัติ
- **ต้องใส่ [break time](/wiki/Beatmap/Break) ภายใต้ข้อจำกัดของ [beatmap editor](/wiki/Client/Beatmap_editor)**
- **[Difficulty settings](/wiki/Client/Beatmap_editor/Song_setup#difficulty) ต้องไม่ใช้ความละเอียดของค่ามากกว่าที่ [beatmap editor](/wiki/Client/Beatmap_editor) รองรับ**
- **การตั้งค่า `Letterbox during breaks` ต้องสอดคล้องกันระหว่าง difficulty ของโหมดเดียวกัน หากมีช่วง break และถ้ามีสตอรีบอร์ดเดียวกันรวมอยู่ด้วย**
- **บีตแมปที่มีสปอยล์สื่ออื่นต้องใส่คำเตือนสปอยล์ในคำอธิบายบีตแมป**<!-- TODO: This and the following rule should be game features, not RC rules. Aside from being ineffective, it doesn't make sense for this to be included in RC because you can edit it after Ranked. -->

### แนวทาง

- **ควรให้เครดิตผู้สร้างสื่อที่ใช้ในบีตแมปไว้ในคำอธิบายบีตแมป** ตาม [Content Usage Guidelines](/wiki/Rules/Content_usage_guidelines) สื่อที่ใช้ในบีตแมปควรเป็นสื่อที่อนุญาตให้ใช้ใน osu! ได้ และไลเซนส์ของสื่อนั้นมักระบุรายละเอียดว่าต้องให้เครดิตผู้สร้างอย่างไร เหตุผลทั่วไปที่อาจไม่ใส่เครดิตในคำอธิบาย เช่น การตามหาแหล่งที่มายากเกินไป ผู้สร้างต้องการไม่ให้เครดิต หรือมีเครดิตอยู่แล้วในช่อง artist, title หรือ source
  - สื่อที่สร้างด้วย AI ในบีตแมปควรถูกระบุอย่างชัดเจนว่าเป็นสื่อที่สร้างด้วย AI ไม่ใช่ภาพวาดหรืออย่างอื่น เพื่อป้องกันการสื่อสารผิดเกี่ยวกับกระบวนการสร้าง
- **ไม่แนะนำให้เอา difficulty Ranked หรือ Loved ของตัวเองมาใช้ซ้ำโดยตรงในบีตแมป Ranked อื่น** เพื่อหลีกเลี่ยงการเพิ่มคอนเทนต์ Ranked โดยไม่จำเป็น
- **ไม่ควรแก้ [slider tick rate](/wiki/Beatmapping/Slider_tick_rate) ผ่านไฟล์ `.osu`** ค่าที่กำหนดเองส่วนใหญ่ทำให้ slider tick ไม่ snap อย่างไรก็ตาม tick rate 0.5, 1.333 และ 1.5 มีการใช้งานจริง และใช้ได้เฉพาะเมื่อไม่ทำให้ slider tick หลุด snap
- **[Kiai](/wiki/Gameplay/Kiai_time) ควรเริ่มบนเสียงในเพลง** หากทำอย่างอื่นจะทำให้แสง kiai รู้สึกไม่เกี่ยวกับเพลง
- **การตั้งค่า `Enable countdown` ควรสอดคล้องกันระหว่าง difficulty ของโหมดเดียวกัน** หาก difficulty ใดมีช่วง intro ไม่ยาวพอให้ countdown ทำงาน การตั้งค่านี้ไม่จำเป็นต้องสอดคล้องกัน

## Beatmap

*หมายเหตุ: [key mode](osu!mania#common-terms) และ [playstyle](osu!mania#common-terms) แต่ละแบบของ [osu!mania](/wiki/Game_mode/osu!mania) ถือเป็นโหมดเกมแยกกันตลอดทั้งส่วนนี้*

### กฎ

- **ทุกโหมดเกมในบีตแมปต้องมี spread ที่เริ่มจากระดับ difficulty ต่ำสุดตาม [drain time](/wiki/Beatmap/Drain_time) ของเพลง** สำหรับ difficulty ที่สูงกว่าระดับต่ำสุดที่กำหนด spread จะข้ามระดับ difficulty ไม่ได้ และห้ามมีช่องว่างความยากที่ใหญ่เกินไประหว่าง difficulty สองอันใด ๆ
- **ทุก difficulty ต้องทำตาม ranking criteria เฉพาะ difficulty ของโหมดนั้น** ดูข้อมูลเพิ่มเติมได้ที่ ranking criteria ของ [osu!](osu!), [osu!taiko](osu!taiko), [osu!catch](osu!catch) และ [osu!mania](/wiki/Ranking_criteria/osu!mania)
- **ทุก difficulty ของบีตแมปต้องมี [drain time](/wiki/Beatmap/Drain_time) อย่างน้อย 30 วินาที**
- **บีตแมปที่รวมเพลง 2 เพลงเข้าด้วยกันต้องทำตามข้อกำหนดขั้นต่ำของ spread ที่สอดคล้องกับความยาวของเพลงที่ยาวกว่า**[^songs-comp-note] เพื่อหลีกเลี่ยงการยืดเพลงแบบจงใจเพื่อเลี่ยงข้อจำกัดด้านเวลาข้างต้น สามารถมีข้อยกเว้นได้สำหรับเพลงที่ปล่อยออกมาคู่กันโดยเฉพาะ
- **ชื่อ difficulty ในบีตแมปต้องแสดงลำดับความยากอย่างชัดเจนและสื่อถึงความยากของตัวเองอย่างถูกต้อง ยกเว้น:**
  - difficulty สูงสุดของแต่ละโหมดเกม
  - difficulty สูงสุดของแต่ละโหมดเกมที่มีระดับความยากใกล้เคียงกัน โดยใช้เฉพาะกับ difficulty ระดับ Insane และ Extra (เช่น difficulty ระดับ Insane ของเซ็ต ENHIIII หรือ difficulty ระดับ Extra ของเซ็ต ENHIIXXX)
- **ชื่อ difficulty แบบกำหนดเองของบีตแมปต้องมีธีมหรือแพตเทิร์นร่วมที่เกี่ยวข้องกับเพลงหรือความยาก และต้องไม่สื่อผิด** ชื่อ difficulty จะถือว่าสื่อผิดหากทำให้เข้าใจว่าเป็นระดับความยากอื่น (เช่น ตั้งชื่อ difficulty "Expert" ว่า "Normal")
- **[beatmap host](/wiki/Beatmap/Beatmap_host) ต้องแมปจำนวน difficulty เท่ากับหรือมากกว่าผู้สร้าง [guest difficulty](/wiki/Beatmap/Guest_difficulty) คนใด ๆ** เพื่อให้เครดิตกับคนที่ควรได้รับ [collaborative difficulty](/wiki/Beatmap/Beatmap_collaborations) จะถือเป็น difficulty บางส่วนเท่านั้น และจะใช้ [drain time](/wiki/Beatmap/Drain_time) เพื่อกำหนดสัดส่วนการมีส่วนร่วมเมื่อ guest แมปมากกว่า host อย่างมีนัยสำคัญ
- **[beatmap host](/wiki/Beatmap/Beatmap_host) และผู้สร้าง [guest difficulty](/wiki/Beatmap/Guest_difficulty) สามารถแก้ไข difficulty ของตัวเองได้ตามต้องการ** หากทั้งสองฝ่ายเห็นไม่ตรงกัน beatmap host ต้องลบผลงานของ guest เมื่อมีการร้องขอ หากติดต่อผู้สร้าง guest difficulty ไม่ได้เป็นเวลา 1 เดือน จะถือว่าอีกฝ่ายยอมรับการเปลี่ยนแปลง
- **บีตแมปห้ามมีส่วนสำคัญของ difficulty ที่เป็นของผู้ใช้อื่นโดยไม่มีการมีส่วนร่วมโดยตรงหรือการอนุญาตอย่างชัดเจนจากเจ้าของ**

### แนวทาง

- **difficulty สูงสุดของบีตแมปควรสอดคล้องกับฟีลโดยรวมของเพลง** difficulty Easy/Normal สามารถใช้เป็น difficulty เดี่ยวของบีตแมปได้ หากจังหวะไม่ได้ถูกทำให้ง่ายเกินไป ไม่เช่นนั้นควรมี difficulty ระดับ Hard ขึ้นไป
- **ชื่อ difficulty ไม่ควรทำให้เข้าใจผิด**  
  - **ชื่อ difficulty แบบบรรยายควรเกี่ยวข้องกับเพลง** เมื่อเพิ่มคำอื่น เช่น `Tragic Love` เข้าไปในชื่อ difficulty คำนั้นควรเกี่ยวข้องกับเพลง
  - **เมื่อนำชื่อผู้ใช้หลายคนมารวมเป็นชื่อ collaborator เดียวสำหรับ [guest difficulty](/wiki/Beatmap/Guest_difficulty) ต้องแน่ใจว่ายังอ่านเข้าใจได้** ตัวอย่างเช่น `Mamayucha` ที่เป็น [collaboration difficulty](/wiki/Beatmap/Beatmap_collaborations) ระหว่าง `Mani`, `Mari`, `Yuzu` และ `Mocha` ทำให้เอกลักษณ์ของผู้ร่วมงานหลายคนหายไปเมื่อเทียบกับการระบุชื่อเต็ม การใช้ `Collab` หรือคำบรรยายอย่าง `osu! Mascots' Expert` อาจชัดเจนกว่า
- **ชื่อผู้ใช้ที่บอกความเป็นเจ้าของ [guest difficulty](/wiki/Beatmap/Guest_difficulty) ควรสอดคล้องกันระหว่างบีตแมปหลายอัน** การใช้ชื่อเล่นหลายแบบสำหรับผู้ใช้คนเดียวทำให้ตีความได้กำกวมหรือทำให้เข้าใจผิดว่าใครเป็นคนสร้าง difficulty
- **หลีกเลี่ยงอักขระ Unicode ที่ไม่ใช่ตัวอักษรหรือตัวเลขในชื่อ difficulty** อักขระเหล่านี้อาจทำให้เกิดข้อผิดพลาดกับ [ระบบส่งบีตแมป](/wiki/Beatmapping/Beatmap_submission) และสร้างปัญหาให้ผู้ใช้บางคนเมื่อแสดงในแชต

## Timing

### กฎ

- **ต้องใช้ [uninherited timing point](/wiki/Client/Beatmap_editor/Timing#uninherited-timing-point) เพื่อแมป time signature ของเพลงให้ถูกต้อง** หาก time signature ที่ไม่ถูกต้องกินเวลามากกว่าหนึ่งห้อง ต้องเพิ่ม uninherited timing point บน downbeat ถัดไปเพื่อรีเซ็ต time signature สำหรับ #/4-signature ที่ editor ไม่รองรับ สามารถใช้ metronome reset หรือแก้ไฟล์ `.osu` ได้ สำหรับ time signature อื่นที่ไม่รองรับ ให้อ้างอิง [ตารางตัวอย่างนี้](/wiki/shared/timing/Timing_signature_reference_chart.png) และดูข้อมูลเพิ่มเติมในไกด์ [Timing songs with #/8-signatures](/wiki/Guides/Timing_songs_with_8-signatures)
- **บีตแมปต้อง timed อย่างสมบูรณ์แบบ** หมายความว่า [BPM](/wiki/Music_theory/Tempo) และออฟเซ็ตของ [uninherited timing point](/wiki/Client/Beatmap_editor/Timing#uninherited-timing-point) แต่ละจุดต้องซิงก์กับเพลงอย่างแม่นยำ บีตแมปที่มี BPM เปลี่ยนตลอดเวลาอาจ timed ให้สมบูรณ์แบบไม่ได้ และควรทำให้แม่นยำที่สุดเท่าที่ทำได้โดยไม่กระทบ gameplay timing ที่ซับซ้อนระหว่าง break หรือสปินเนอร์เป็นสิ่งที่ทำหรือไม่ทำก็ได้
- **[Uninherited timing point](/wiki/Client/Beatmap_editor/Timing#uninherited-timing-point) ต้องเหมือนกันในทุก difficulty ของบีตแมป** แต่ละจุดต้องมี [BPM](/wiki/Music_theory/Tempo) และออฟเซ็ตเหมือนกันในทุก difficulty
- **ควรใช้ [uninherited timing point](/wiki/Client/Beatmap_editor/Timing#uninherited-timing-point) เฉพาะเมื่อจำเป็นต่อ timing เท่านั้น** timing point ที่ไม่จำเป็นอาจส่งผลต่อการกระพริบในเมนูหลักโดยไม่ตั้งใจ เพิ่มเสียงที่ไม่ต้องการให้ [ม็อด Nightcore](/wiki/Gameplay/Game_modifier/Nightcore) หรือทำให้ timing เคลื่อนได้ ตัวอย่างการใช้ที่ยอมรับได้ ได้แก่:
  - จัด beat ของม็อด Nightcore ให้ตรงกับจุดเริ่มต้นของท่อนเพลง
  - รองรับ object ในท่อนเพลงที่ต้องใช้ beat snap divisor ที่ไม่รองรับ (เช่น 1/11)
- **ห้ามวาง [uninherited](/wiki/Client/Beatmap_editor/Timing#uninherited-timing-point) สองจุด หรือ [inherited timing point](/wiki/Client/Beatmap_editor/Timing#inherited-timing-point) สองจุดไว้ตำแหน่งเดียวกัน** การมี uninherited หรือ inherited timing point สองจุดซ้อนกันจะทำให้ slider velocity และการตั้งค่า volume ทำงานผิดจากที่ตั้งใจ
- **ห้ามวาง [inherited timing point](/wiki/Client/Beatmap_editor/Timing#inherited-timing-point) ก่อน [uninherited timing point](/wiki/Client/Beatmap_editor/Timing#uninherited-timing-point) จุดแรก** หากไม่มีการตั้งค่าให้สืบทอด inherited timing point จะทำงานไม่ถูกต้อง หากต้องการเปลี่ยน hitsound หรือ slider velocity ก่อน uninherited timing point จุดแรก ต้องเลื่อนจุดนั้นถอยไปหนึ่ง measure เต็มเพื่อให้สามารถใช้ inherited timing point ได้
- **[Uninherited timing point](/wiki/Client/Beatmap_editor/Timing#uninherited-timing-point) จุดแรกของบีตแมปห้ามใช้เปิด/ปิด [kiai](/wiki/Gameplay/Kiai_time)** การทำแบบนี้จะทำให้ kiai กะพริบก่อน object ปรากฏ ต้องใช้ [inherited point](/wiki/Client/Beatmap_editor/Timing#inherited-timing-point) ที่ตำแหน่งเดียวกับ uninherited point จุดแรกแทน
- **Hit object ต้อง snap อยู่ห่างจาก tick ใด ๆ บนไทม์ไลน์น้อยกว่า 2 ms** [AiMod](/wiki/Client/Beatmap_editor/AiMod) จะรายงานปัญหาเหล่านี้ รวมถึง false positive ที่พบได้ยาก false positive มักเกิดที่ท้ายสไลเดอร์และ reverse และควรตรวจยืนยันด้วยตัวเองหรือเครื่องมืออื่น Object ในท่อนเพลงที่ต้องใช้ beat snap divisor ที่ไม่รองรับ (เช่น 1/11) สามารถ:
  - ปล่อยให้ไม่ snap ได้ ตราบใดที่ตรงกับ beat snap divisor ที่ตั้งใจไว้
  - snap ผ่านการเปลี่ยน [BPM](/wiki/Music_theory/Tempo) ชั่วคราว
- **Object ที่ snap ผิดเพราะผ่านหรือจบก่อน [uninherited timing point](/wiki/Client/Beatmap_editor/Timing#uninherited-timing-point) ใหม่เล็กน้อย ต้อง snap ส่วนท้ายของมันให้อยู่ใน timing section ใหม่** สำหรับสปินเนอร์และ long note ของ osu!mania ทำได้โดยลากท้าย object ในไทม์ไลน์ สำหรับสไลเดอร์ ทำได้ผ่านการปรับ slider velocity หรือแก้ไฟล์ `.osu`

## Audio

### กฎ

- **ไฟล์เสียงของบีตแมปต้อง...**
  - **...เข้ารหัสเป็น MP3 (`.mp3`) หรือ [Ogg Vorbis](https://en.wikipedia.org/wiki/Vorbis) (`.ogg`)**
  - **...มี average bit rate ไม่เกิน 192 kbps สำหรับไฟล์ MP3 หรือ 208 kbps สำหรับไฟล์ Ogg Vorbis**
  - **...มี sampling rate ไม่เกิน 48 kHz**
  - **...มี average bit rate ไม่ต่ำกว่า 128 kbps** หากมีแหล่งไฟล์แบบนั้นอยู่ ไม่เช่นนั้นให้ใช้คุณภาพสูงสุดที่หาได้
  - **...ห้ามถูกอัป encode จาก bitrate หรือ sampling rate ที่ต่ำกว่า**
- **บีตแมปหนึ่งอันมีไฟล์เพลงที่ใช้ร่วมกันทุก difficulty ได้เพียงไฟล์เดียว** การมีไฟล์เพลงหลายไฟล์ในบีตแมปเดียวไม่รองรับ และทำให้เกิดพฤติกรรมที่ไม่คาดคิดกับ preview time, metadata ฯลฯ
- **บีตแมปต้องมี [hitsound](/wiki/Beatmapping/Hitsound)** hitnormal ให้ feedback แก่ผู้เล่น ส่วน addition (whistle, clap และ finish) ช่วยเน้นส่วนสำคัญของเพลง[^normal-vs-addition]
  - **บีตแมป osu!mania ไม่จำเป็นต้องมี hitsound addition**[^normal-vs-addition] เพื่อให้แมปเปอร์ osu!mania จากพื้นฐานที่ต่างกันเข้าถึงได้ง่ายขึ้น อย่างไรก็ตาม ยังแนะนำอย่างยิ่งให้ใช้ hitsound addition เพื่อเพิ่มฟีลให้บีตแมป
- **ทุกส่วนของ object ที่ต้องคลิกต้องมี hitsound อย่างน้อยหนึ่งเสียงที่ทั้ง...**
  - **...มี impact ชัดเจน โดย peak ของเสียงดีเลย์ไม่เกิน 5 มิลลิวินาที** ยกเว้น `normal-hitfinish.wav` จากสกิน default
  - **...ใช้ไฟล์รูปแบบ WAV แบบไม่บีบอัด (`.wav`) หรือ Ogg Vorbis (`.ogg`)** ไม่ควรใช้ MP3 ตรงนี้ เพราะมีดีเลย์โดยธรรมชาติ
  - สิ่งนี้ช่วยให้มีเสียง feedback ทันทีเมื่อคลิก object และซิงก์กับเพลงในแบบที่ผู้เล่นใช้ตัดสินได้ว่าคลิกเร็วหรือช้าเกินไป osu!mania ได้รับการยกเว้นจากข้อนี้เพราะมี object ที่เกิดพร้อมกัน
- **Hitsound ต้องได้ยินชัด** จุดประสงค์ของมันคือการให้ feedback ดังนั้น hitsound ที่ volume ต่ำมาก หรือ sample ที่กลืนกับ sample ของเพลงจึงไม่ยอมรับได้ โหมดเกมเฉพาะจะระบุข้อยกเว้นของกฎนี้ไว้ใน ranking criteria ของแต่ละโหมด
- **ต้องตั้ง preview point และต้องสอดคล้องกันระหว่างทุก difficulty ของบีตแมป** ใช้ทั้งในเมนูเลือกเพลงและ preview thumbnail ออนไลน์
- **ไฟล์ hitsound ทุกไฟล์ต้องยาวอย่างน้อย 25ms** ไฟล์ที่สั้นกว่านั้นอาจทำให้ไม่มีเสียงเล่นในเกม
- **ไฟล์เสียงที่เงียบสนิทต้องใช้ [ไฟล์ 44 ไบต์นี้](https://assets.ppy.sh/media/blank.wav)** ไฟล์อื่นมีขนาดใหญ่เกินจำเป็น และไฟล์ 0 ไบต์จะใช้งานไม่ได้
- **[Storyboarded hitsound](/wiki/Beatmapping/Hitsound#storyboarded-hitsound) ห้ามใช้แทน [active hitsound](/wiki/Beatmapping/Hitsound#active-hitsound)** เพราะให้ feedback แก่ผู้เล่นไม่ถูกต้อง Storyboarded hitsound ในสถานการณ์อื่นยอมรับได้ แต่ไม่แนะนำ osu!mania ได้รับการยกเว้นจากกฎนี้

### แนวทาง

- **ไฟล์เสียงและไฟล์ hitsound ของบีตแมปไม่ควรมีเสียงเพี้ยนที่ได้ยินชัดและไม่สมเหตุสมผล** เช่น clipping, muffling หรือ crackling ที่ชัดเจนว่าไม่ได้ตั้งใจโดยศิลปินหรือไม่ได้เป็นส่วนหนึ่งของเอกลักษณ์เพลง วิธีตัดสินที่ดีที่สุดคือฟังเสียงจริง ไม่ใช่พึ่งซอฟต์แวร์อย่างเดียว
- **ไฟล์เสียงของเพลงไม่ควรถูกยืดแบบเทียมเพื่อให้ผ่านข้อจำกัดด้านเวลาในส่วนบีตแมปของ criteria นี้** ซึ่งรวมถึงแต่ไม่จำกัดแค่การ loop บางท่อนของไฟล์เสียง การลด [BPM](/wiki/Music_theory/Tempo) ของเพลงหรือท่อนเพลง หรือการเพิ่มดนตรีสั้น ๆ เข้าไปโดยไม่ผสานเข้ากับทั้งเพลง ข้อนี้ไม่ใช้กับ [song compilation](/wiki/Beatmap/Song_compilation) หรือไฟล์เสียงที่สั้นกว่าความยาวบีตแมปขั้นต่ำที่จัดอันดับได้
- **หากคุณไม่ได้แมป 20% ท้ายสุดของไฟล์เสียงบีตแมป ควรตัดส่วนนั้นออก**
  - ข้อนี้ไม่ใช้ หาก...
    - outro ถูกใช้โดยวิดีโอหรือสตอรีบอร์ด
    - การแมป outro จะทำให้ gameplay แย่ เช่น fade-out หรือปัญหา timing
- **[Song compilation](/wiki/Beatmap/Song_compilation) หรือแทร็กสไตล์เมดเลย์อื่น ๆ ควรพยายามมิกซ์ให้สะอาด มี transition ที่เหมาะสม และไม่ควรมีการตัดพักแบบฉับพลันหรือ fade ยาวระหว่างเพลงต่าง ๆ** เพลงที่ใช้ใน compilation ควรมีคุณภาพเสียง volume และความยาวใกล้เคียงกัน เพื่อให้ compilation ได้ประสบการณ์ gameplay ที่กลมกลืนเหมือนบีตแมปอื่น transition แบบ cross-fade ยอมรับได้ แต่ควรใช้เท่าที่จำเป็นและโดยทั่วไปไม่ควรยาวเกิน 5 วินาที transition ที่ match beat และวางแผนมาดีมักเป็นตัวเลือกที่ดีกว่า cross-fade เสมอ
- **การรวมเพลง 2 เพลงควรมีความเกี่ยวข้องกันอย่างชัดเจนและใกล้ชิด** ตัวอย่างเช่น แต่ไม่จำกัดแค่ เป็นเพลงในซีรีส์เดียวกันคนละเวอร์ชัน เกี่ยวข้องกันทางเนื้อร้องหรือ motif มีโทนและ/หรือแนวเพลงคล้ายกัน ฯลฯ
- **เพลงที่ถูกตัดควรรักษาภาพรวมและความเข้มข้นของเพลงเต็มไว้** การตัดที่เปลี่ยนโครงสร้างของเพลงเต็ม (เช่น ตัด intro/verse/chorus/outro ออก หรือจัดเรียงใหม่) อาจทำให้สื่อเพลงผิดไปและมักทำให้ประสบการณ์การเล่นไม่น่าพอใจ ข้อนี้ไม่ใช้กับ official cut หรือการทำซ้ำ official cut
- **[Gameplay sound](/wiki/Skinning/Sounds#gameplay) ที่ไม่ใช่ [active hitsound](/wiki/Beatmapping/Hitsound#active-hitsound) ควรใช้ไฟล์ MP3 (`.mp3`) หรือ Ogg Vorbis (`.ogg`)** ไฟล์เหล่านี้มักมีระยะเวลายาว และไฟล์ WAV แบบไม่บีบอัด (`.wav`) มีขนาดใหญ่เกินจำเป็นเมื่อเทียบกัน อย่างไรก็ตาม ควรใช้ WAV แบบไม่บีบอัดเมื่อทำให้ไฟล์มีขนาดเล็กกว่า
- **หลีกเลี่ยงการแทนที่ hit finish ใน sampleset soft/normal ด้วย sample hitsound custom ที่ใช้บ่อย** การใช้ finish เหล่านี้แทนเสียง snare/bass drum หรือเมโลดี้ของเพลงอาจฟังน่ารำคาญสำหรับคนที่ปิด beatmap hitsounds แนะนำให้แทนที่ hit whistle/clap เพราะ sample เหล่านั้นถูกใช้บ่อยกว่า บีตแมป osu!taiko ได้รับการยกเว้นจากแนวทางนี้ และมีแนวทาง hitsound sample เฉพาะโหมดของตัวเอง

## Video and background

### กฎ

- **ต้องมีภาพพื้นหลังในทุก difficulty ของบีตแมป** สามารถใช้ไฟล์พื้นหลังต่างกันในแต่ละ difficulty ได้

  - **ความกว้างขั้นต่ำ:** 160px
  - **ความสูงขั้นต่ำ:** 120px

  - **ความสูงสูงสุด:** 1440px[^maximum-dimensions]

- **ขนาดของวิดีโอต้องไม่เกินความกว้าง 1280 และความสูง 720 พิกเซล** นอกจากนี้ควรหลีกเลี่ยงการ upscale วิดีโอความละเอียดต่ำให้เป็นความละเอียดสูง เพื่อไม่ให้ไฟล์วิดีโอใหญ่หรือกินทรัพยากรมากเกินไป
- **วิดีโอต้องเข้ารหัสเป็น H.264**
- **ออฟเซ็ตของวิดีโอต้องถูกต้อง หากวิดีโอซิงก์กับเพลง** ออฟเซ็ตที่ผิดอาจทำให้ภาพสื่อเพลงผิดไป หากวิดีโอเดียวกันปรากฏในหลาย difficulty ต้องใช้ออฟเซ็ตเดียวกันเสมอ
- **ต้องลบ audio track ออกจากไฟล์วิดีโอ** audio track ในวิดีโอไม่ได้ถูกใช้ใน osu! และการลบออกช่วยลดขนาดไฟล์ของบีตแมป รวมถึงวิดีโอที่มี audio track แบบปิดเสียงด้วย
  - *ดูแนวทางได้ที่ [Compressing files § Video](/wiki/Guides/Compressing_files#video)*

### แนวทาง

- **ภาพพื้นหลังและวิดีโอในบีตแมปควรมีคุณภาพสมเหตุสมผล** พยายามหาแหล่งต้นฉบับและหลีกเลี่ยงการ upscale หรือทำให้ไฟล์ใหญ่โดยไม่จำเป็น

## Skinning

### กฎ

- **หากคุณใช้องค์ประกอบที่สร้างโดยสมาชิกชุมชนคนอื่น ต้องขออนุญาตก่อน** การเคารพผลงานของคนอื่นเป็นเรื่องสำคัญที่สุด และคนส่วนใหญ่ยินดีที่ผลงานของพวกเขาถูกนำไปใช้ในโปรเจกต์ของคุณ ดังนั้นหากคุณไม่รู้ว่าใครเป็นคนสร้างองค์ประกอบที่คุณจะใช้ คุณต้องไม่ใช้มัน
- **[Gameplay element](/wiki/Ranking_criteria/Skin_set_list) ต้องมองเห็นได้** คุณห้ามทำให้องค์ประกอบใด ๆ ที่กระทบความสามารถในการเล่นของบีตแมปล่องหน เพราะจะทำให้บีตแมปเล่นไม่เป็นธรรมชาติหรือถึงขั้นเล่นไม่ได้ (`cursormiddle.png` เป็นข้อยกเว้น เพราะส่งผลต่อพฤติกรรมของ cursor trail) องค์ประกอบที่ไม่เกี่ยวกับ gameplay สามารถโปร่งใสได้เฉพาะเมื่อมีเหตุผลที่ถูกต้อง และการกระทำนั้นไม่ส่งผลเสียต่อการใช้งาน interface element
- **องค์ประกอบที่สกินต้อง crop ให้สะอาด ไม่ให้มี artifact เป็นพิกเซลรอบ ๆ หรือเงาที่ถูกตัดครึ่ง**
- **องค์ประกอบที่สกินห้ามมีขนาดใหญ่จนทับองค์ประกอบสกินอื่นที่ปกติจะไม่ทับกันในสกิน default** ข้อนี้ใช้เฉพาะส่วนที่มองเห็นได้ของภาพ ซึ่งอาจทำให้ประสบการณ์ gameplay บิดเบือนไปโดยบังองค์ประกอบที่ปกติควรมองเห็น
- **เมื่อสกิน [gameplay element](/wiki/Ranking_criteria/Skin_set_list) ต้องสกิน element ให้ครบทั้งชุด เพื่อหลีกเลี่ยงความขัดแย้งระหว่างสกินเฉพาะผู้ใช้กับสกินเฉพาะบีตแมป** เมื่อสกิน element ที่ถูกระบุว่า optional คุณต้องใส่ element ที่ required ทั้งหมดของชุดนั้น แต่สามารถข้าม optional element อื่นได้ เว้นแต่ element เหล่านั้นจะถูกจัดกลุ่มร่วมกับ element ที่คุณกำลังสกิน อย่างไรก็ตาม หาก required skin element ในชุดนั้นจะไม่ถูกใช้งาน หรือมีการบังคับใช้สกิน default ก็ไม่จำเป็นต้องใส่ element นั้น

### แนวทาง

- **องค์ประกอบที่สกินควรเก็บเป็นไฟล์ `.png` หากใช้ความโปร่งใส** หากไม่ได้ใช้ความโปร่งใส สามารถใช้รูปแบบไฟล์ใดก็ได้ที่มีขนาดเล็กที่สุดและรองรับการสกินใน osu!

## Storyboarding

### อภิธานศัพท์

- **Storyboard image:** หมายถึงรูปภาพในโฟลเดอร์เพลงที่สตอรีบอร์ดใช้
- **Sprite:** object ในสตอรีบอร์ดที่แทนรูปภาพหรือชุดรูปภาพ
- **Time:** ตำแหน่งบนไทม์ไลน์ในหน่วยมิลลิวินาที การแสดงผลแบบนี้เห็นได้ในส่วน design ของ editor
- **Command:** สิ่งที่ส่งผลต่อ sprite ได้หลายแบบ ตัวอย่าง command เช่น `Move`, `Scale`, `Fade` และ `Rotate` แต่ละ command มี starttime และ endtime
- **Axis-specific command:** command ที่ใช้กับแกนพื้นที่ที่กำหนดไว้เพียงแกนเดียว เช่น `MoveX` และ `MoveY`
- **Active:** ช่วงตั้งแต่ start time แรกจนถึง end time สุดท้ายของ command ใน object
- **Rendered:** มักหมายถึง sprite บนหน้าจอที่ยังไม่ fade out จนหมด
- **osu!pixel:** หน่วยมิติที่เล็กที่สุดของแท็บ design เห็นได้ที่มุมขวาบนของหน้าจอ editor เช่น `x: 104; y: 88`

### กฎ

- **ภาพที่ใช้ในสตอรีบอร์ดต้องมีพื้นที่ไม่เกิน 17,000,000 พิกเซล เพื่อให้เวลาโหลดของภาพขนาดใหญ่อยู่ในระดับที่คอมพิวเตอร์ส่วนใหญ่รับไหว** นอกจากนี้ คุณอาจต้องปรับขนาดภาพให้เหมาะกับขนาดสูงสุดภายในของ storyboard editor ที่ 854 x 480 osu!pixels เมื่อนำไปใช้
- **Difficulty ต้องไม่เกิด parsing error ตอนโหลด** หมายความว่า parser ต้องอ่านคำสั่งสตอรีบอร์ดทุกส่วนได้
- **การตั้งค่า `Widescreen support` ต้องสอดคล้องกันระหว่าง difficulty ที่มีสตอรีบอร์ดในบีตแมปเดียวกัน** เว้นแต่สตอรีบอร์ดเฉพาะ difficulty ถูกออกแบบมาสำหรับอัตราส่วนภาพที่ต่างกัน

### แนวทาง

- **ไม่ควรมี sprite และ command ที่ยัง active อยู่หลังจบเพลง** ข้อนี้ยืดหยุ่นได้อีกไม่กี่วินาทีตามเอฟเฟกต์ของสตอรีบอร์ด แต่ไม่ควรมากกว่านั้น
- **ควรเว้นขอบโปร่งใส 1 พิกเซลรอบภาพสตอรีบอร์ดของ sprite ที่ถูกหมุน เพื่อให้ interpolation ทำงานถูกต้อง** osu! ไม่ใช้ anti-aliasing รอบภาพ ดังนั้นจะเห็นได้ชัดมากหากขอบภาพมองเห็นและ sprite ถูกหมุน
- **หลีกเลี่ยงปัญหา performance ที่สังเกตได้ให้มากที่สุด แม้จะ optimized แล้วก็ตาม frame rate ที่สม่ำเสมอเป็นสิ่งสำคัญต่อประสบการณ์การเล่นของ difficulty** ให้ test play difficulty ระหว่างขั้นตอนม็อดดิ้งเพื่อยืนยันเรื่องนี้
- **หลีกเลี่ยงการใช้ sound sample ของสตอรีบอร์ดในแบบที่ทำให้สับสนกับ hitsound ระหว่าง gameplay ได้ง่าย** สิ่งนี้ขัดกับแนวคิดของเสียง feedback เพราะ sound sample จะเล่นโดยไม่ขึ้นกับ input ใด ๆ จากผู้เล่น
- **หลีกเลี่ยง command ที่ไม่สมเหตุสมผล ขัดแย้งกัน หรือเลิกใช้แล้ว** command ที่มี ending time ก่อน start time หรือผูกกับ trigger ที่ไปถึงไม่ได้ จะไม่ทำงานตามตั้งใจหรือกลายเป็นของที่ไม่ใช้งาน ควรถูกลบหรือปรับให้ทำงานตามตั้งใจ command ประเภทเดียวกันที่มีช่วงเวลาซ้อนกันควรปรับช่วงเวลาและพารามิเตอร์ให้ไม่ซ้อนกันอีก
- **ควรเปิดการตั้งค่า `Widescreen support` หาก difficulty มีสตอรีบอร์ดแบบ widescreen** อีกทางหนึ่ง หากสตอรีบอร์ดออกแบบมาสำหรับความละเอียด 4:3 ควรปิด widescreen support การตั้งค่านี้จะไม่ส่งผลอะไรใน difficulty ที่ไม่มีสตอรีบอร์ด
- **ตรวจให้แน่ใจว่าสตอรีบอร์ดถูก optimize มากที่สุดเท่าที่ทำได้** ภายใต้ความเหมาะสมในทางปฏิบัติ
  - **หลีกเลี่ยงการมี sprite หรือพื้นหลังของ difficulty ที่ถูกบังจนมองไม่เห็นทั้งหมดในขณะที่ยัง rendered อยู่** การ fade out ออกไปเมื่อไม่เห็นอยู่แล้วจะดีกว่าสำหรับ performance หากต้องการ fade out พื้นหลังของ difficulty ให้เปลี่ยนภาพพื้นหลังเดียวกันเป็น sprite โดยใช้ `Background` หรือ `0` เป็นพารามิเตอร์ที่สอง แล้ว fade ตามต้องการ
  - **หลีกเลี่ยง sprite ที่อยู่นอกจอบางส่วนหรือถูกบังตลอดเวลาที่ใช้งาน** ในกรณีเหล่านี้ควรตัดส่วนที่เกี่ยวข้องของภาพออก เว้นแต่จำเป็นต่อเอฟเฟกต์ในสตอรีบอร์ด
  - **หลีกเลี่ยงพื้นที่โปร่งใสที่ไม่จำเป็นรอบภาพสตอรีบอร์ด** เพื่อ performance ควร crop ภาพให้มากที่สุดเท่าที่เอฟเฟกต์ต้องการ
  - **ใช้ loop สำหรับ command ที่ทำซ้ำหลายครั้ง เว้นแต่จะขัดกับภาพที่ตั้งใจไว้** การใช้ loop command มักลดจำนวนบรรทัดได้มาก ซึ่งช่วยลดขนาดไฟล์
  - **หลีกเลี่ยงการใช้ axis-specific command สองคำสั่งเมื่อสามารถทำเอฟเฟกต์เดียวกันได้ด้วย regular command คำสั่งเดียว** การใช้หนึ่ง command แทนสอง command จะทำให้ขนาดไฟล์โดยรวมเล็กลง
  - **ใช้รูปแบบไฟล์ภาพที่มีขนาดเล็กที่สุดโดยยังรักษาคุณภาพที่สมเหตุสมผลไว้** ไฟล์ `.png` มักใช้พื้นที่มากกว่าสำหรับภาพขนาดใหญ่เพราะเป็นการบีบอัดแบบ lossless ต่างจาก `.jpg`
  - **หลีกเลี่ยงไฟล์ภาพซ้ำ** การมีภาพเดียวกันแบบเป๊ะ ๆ สองไฟล์เพิ่มขนาดไฟล์โดยไม่จำเป็น
  - **หลีกเลี่ยงการมี sprite หลายตัว active อยู่ทั้งที่ไม่ได้ rendered** sprite ที่ active อยู่จะยังประมวลผล command ไม่ว่าจะมองเห็นหรือไม่ หากเกิดกรณีนี้เป็นเวลานาน ควรสร้าง sprite ใหม่เมื่อกลับมาเห็นอีกครั้งแทน
  - **เมื่อใช้ command ประเภทเดียวกันจำนวนมากบน sprite ให้พยายามเว้นอย่างน้อย 16 ms ระหว่าง start time ของแต่ละคำสั่ง** 60 command ต่อวินาทีมักมากเกินพอสำหรับ sprite ใด ๆ ที่ต้องการ transition ลื่นบนเครื่องทั่วไป ข้อนี้ช่วยลดขนาดไฟล์และเวลาโหลด
  - **Fade out sprite ที่ถูกเปิดใช้งานจาก trigger หลังใช้งานเสร็จ** trigger จะเริ่มทำงานจาก command แรกที่เป็นไปได้และ active ต่อไปจนจบ difficulty ดังนั้นควร fade out เมื่อใช้งานเสร็จ
